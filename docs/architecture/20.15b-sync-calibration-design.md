## 1. Modelo mental do mecanismo atual

O sync local↔nuvem vive em `cloudLoadAll`/`syncFromCloud` (`gdp-core.js`). Por chave (`row.key`), a reconciliação passa por uma **cadeia de `continue`** — qualquer guarda que dispara aborta a sobrescrita daquela chave. Peças relevantes:

- **`getDataTimestamp(data, fallback)`** (L309): lê `updatedAt`/`updated_at`; retorna epoch ms ou `0`. **É a base de toda decisão temporal** e já existe — vamos usá-la como árbitro.
- **`mergeArraysPreservingItens(local, cloud)`** (L317): faz merge por `id`, mantendo a entry com mais `.itens`. É a guarda do AC2 — **deve ser preservada**.
- **`_gdpBootInProgress`** (L584): inicia `true`, desligado por `setTimeout(…3000)` em `gdp-init.js:2999`. Bloqueia saves em 3 pontos (L243, L586, L1013).

### Os 5 portões (estado atual)

| # | Linha | Condição que ABORTA a sobrescrita | Problema |
|---|-------|-----------------------------------|----------|
| 1 | L460-463 | `localHasContent && !cloudHasMoreContent` (`cloudItems >= localItems`) | **Absoluto** — ignora timestamp. Local com +1 entry trava a nuvem para sempre. |
| 2 | L466-480 | `localDeepItens > 0 && !cloudHasMoreDeepContent` → merge | Merge é bom (AC2), mas dispara mesmo com nuvem mais nova legítima. |
| 3 | L482-487 | `msSinceLocalSave < 5000` (dirty window) | **Cego** — não distingue digitação do usuário de save de boot/sync. |
| 4 | L492-494 | `shouldWrite` exige 3 condições de contagem simultâneas | Empate de contagem (120==120) nunca satisfaz → nuvem nova não vence. |
| 5 | L243/586/1013 + init:2999 | `_gdpBootInProgress` por 3s fixos | Alterações nos primeiros 3s não sobem; janela arbitrária. |

**Diagnóstico arquitetural:** o sistema foi calibrado por **contagem de itens** (proxy de "completude") como árbitro primário, com timestamp só como desempate secundário. Isso protege contra cloud-zerado, mas causa a divergência crônica quando duas réplicas têm a mesma contagem com conteúdos diferentes.

---

## 2. Princípio de design (decisão arquitetural central)

> **Inverter a hierarquia de arbitragem: o timestamp passa a ser o árbitro primário; a contagem vira guarda de segurança condicional.**

Regra mestra:
- **A nuvem vence quando `cloudTime > localTime`** (é comprovadamente mais recente), **EXCETO** quando isso destruiria conteúdo local legítimo — caso em que faz-se **merge** (não bloqueio cego).
- **O local é mantido** quando `localTime >= cloudTime` (local é igual ou mais novo).
- **Cloud-zerado** (`cloudItems === 0 && localItems > 0`) **sempre** preserva o local, independentemente do timestamp (guarda dura AC3).

Isso converge as réplicas (AC1/AC6) sem reabrir o risco que os portões originais fecharam (AC2/AC3).

## 3. Decisões por portão (especificação para o @dev)

### Portão 1 (L460-463) — condicionar ao timestamp
**De:** bloqueio absoluto se local tem mais entries.
**Para:** só bloquear se local tem mais entries **E** o local não é mais antigo que a nuvem.
```
// ANTES: if (localHasContent && !cloudHasMoreContent) continue;
// DEPOIS (conceitual):
if (localHasContent && !cloudHasMoreContent && cloudTime <= localTime) continue;
// Se cloudTime > localTime, NÃO aborta aqui — segue para o merge do portão 2,
// que preserva entries locais órfãs (não presentes na nuvem) via mergeArraysPreservingItens.
```
> `cloudTime`/`localTime` já são calculados em L441-442 — basta movê-los para ANTES do portão 1 (hoje estão depois). **Ação:** subir as 2 linhas de cálculo de timestamp para logo após L443.

### Portão 2 (L466-480) — merge passa a ser o caminho normal quando nuvem é mais nova
**Manter** `mergeArraysPreservingItens` (guarda AC2 intacta). Ajuste: quando `cloudTime > localTime`, o merge deve **priorizar o conteúdo da nuvem por entry**, ainda preservando entries locais que a nuvem não tem (órfãs locais). A função atual já faz "mantém local só se tem mais `.itens`" — isso é seguro e pode permanecer. **Nenhuma mudança na função**; só garantir que esse ramo seja alcançável quando a nuvem é mais nova (consequência do ajuste do portão 1).

### Portão 3 (L482-487) — dirty window só para edição ativa REAL
**Resolução do AC4 (o ponto vago que o @po sinalizou):**

Critério objetivo de "edição ativa do usuário" = **o último save local daquela chave foi disparado por uma ação de UI do usuário, não por boot/sync/migração.**

Implementação proposta — marcar a origem do save:
```
// Novo: registrar a ORIGEM do último save, além do timestamp.
// _lastLocalSave[key] já guarda o timestamp (L984/991). Adicionar:
var _lastLocalSaveOrigin = {};   // key -> 'user' | 'system'
// saveWrappedArray ganha 2º parâmetro opcional origin='system' (default).
// As funções de save chamadas por handlers de UI (ex.: salvarPedido, editar*,
// excluir*) passam origin='user'. Boot/sync/migração usam o default 'system'.

// Dirty window passa a checar:
const recentUserEdit = (Date.now() - getLastLocalSave(row.key) < 5000)
                       && _lastLocalSaveOrigin[row.key] === 'user';
if (recentUserEdit) continue;   // protege só digitação real
```
> Isso preserva a proteção contra sobrescrever o que o usuário está editando AGORA, mas para de travar dados antigos só porque o boot salvou algo há <5s. **Escopo:** marcar `origin:'user'` apenas nos handlers de escrita de UI — não precisa cobrir todos os saves, o default seguro é `'system'`.

### Portão 4 (L492-494) — timestamp como condição primária
**De:** `(isSharedKey && cloudHasMoreContent && cloudHasMoreDeepContent) || (cloudTime > localTime && cloudHasMoreDeepContent) || (!localTime && cloudTime === 0)`
**Para (conceitual):**
```
const cloudIsNewer = cloudTime > localTime;
const cloudNotDestructive = cloudHasMoreDeepContent;        // não perde itens nested
const shouldWrite = useTimestampOnly
  ? (cloudTime > localTime || (!localTime && cloudTime === 0))
  : (
      (cloudIsNewer && cloudNotDestructive)                 // ← NOVO: nuvem nova + não-destrutiva vence
      || (isSharedKey && cloudHasMoreContent && cloudHasMoreDeepContent)  // mantém regra antiga
      || (!localTime && cloudTime === 0)
    );
```
> Quando a nuvem é mais nova mas teria menos `.itens` nested, NÃO entra aqui — cai no merge do portão 2 (preserva). Isso satisfaz AC1 (empate de contagem + nuvem nova vence) sem violar AC2.

### Portão 5 (`_gdpBootInProgress`, init:2999) — janela determinística, não cega
**Problema:** 3s fixos bloqueiam saves; alterações nesse intervalo somem.
**Resolução:** em vez de bloquear o save e descartá-lo, **enfileirar** alterações feitas durante o boot e **flush** ao desligar a flag.
```
// saveWrappedArray, quando _gdpBootInProgress: em vez de 'return' seco,
// registrar a chave em um Set _pendingBootSaves.
// Em gdp-init.js, ao setar _gdpBootInProgress=false (timeout 3000),
// disparar um flush: para cada key em _pendingBootSaves, refazer o save (agora sobe).
```
> Mantém AC7 (nada bloqueante no boot — o flush é assíncrono e pós-boot) e cumpre AC5 (alterações do boot sobem). **Alternativa mais simples** se o flush for complexo: reduzir a janela só ao tempo real de carga (desligar a flag ao fim do `loadData()` em vez de timeout fixo) — porém o enfileiramento é mais robusto.

## 4. Ordem de implementação segura (uma mudança reversível por vez)

1. **Mover cálculo de `cloudTime`/`localTime`** para antes do portão 1 (refactor sem efeito comportamental — valida que nada quebra).
2. **Portão 4** (condição `shouldWrite`) — a mudança de maior impacto em AC1; testar isolada.
3. **Portão 1** (condicionar ao timestamp) — depende do passo 1.
4. **Portão 3 + AC4** (origem do save) — mais invasiva (toca handlers de UI); fazer por último no `cloudLoadAll`.
5. **Portão 5** (flush de boot) — independente; pode ser feito em paralelo lógico, testar sozinho.
6. **Bump** `gdp-core.js?v=N` no HTML.

Cada passo deve ser **commit isolado** para permitir rollback cirúrgico (lição do incidente 2026-06-16).

## 5. Riscos arquiteturais e mitigações

| Risco | Severidade | Mitigação |
|-------|-----------|-----------|
| Inverter arbitragem zera dados se timestamps forem não-confiáveis | 🔴 CRÍTICO | `getDataTimestamp` retorna 0 para datas inválidas → cai nas guardas de contagem (comportamento atual). Guarda AC3 (cloud-zerado) é absoluta. |
| Marcar `origin:'user'` incompleto deixa edição desprotegida | 🟡 MÉDIO | Default `'system'` é o lado seguro (não bloqueia); pior caso = comportamento atual. Cobrir os handlers de escrita principais (pedidos, notas, contas). |
| Flush de boot reintroduz saves em massa (incidente!) | 🔴 CRÍTICO | Flush é pós-boot, assíncrono, deduplicado por chave (Set). Teste de carga real obrigatório. |
| Merge produz duplicatas | 🟡 MÉDIO | `mergeArraysPreservingItens` já deduplica por `id`; não alterada. |

## 6. Plano de teste (gate de QA — obrigatório antes de deploy)

Cobertura dos ACs com dados reais:
1. **AC1:** 2 réplicas, 120==120 notas com 1 distinta, nuvem mais nova → converge.
2. **AC2:** pedido local com +itens, local mais novo → não perde.
3. **AC3:** nuvem retorna 0 → local preservado.
4. **AC4:** digitar num campo (origin user) + sync → não sobrescreve; save de boot + sync → sobrescreve.
5. **AC5:** alterar dado <3s após boot → confirma upload ao Supabase.
6. **AC6:** 2 navegadores reais convergem sem Forçar Sync.
7. **AC7:** carga 120 notas / 127 pedidos → sem travamento.

**Ambiente:** testes 1-5 em homologação/local; 6-7 exigem produção real + 2 máquinas (sessão do usuário).
