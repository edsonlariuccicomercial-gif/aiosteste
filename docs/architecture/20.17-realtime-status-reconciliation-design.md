## 1. Modelo mental do mecanismo atual

Há **dois** subsistemas de sync independentes neste app, que não devem ser confundidos:

- **Boot/pull sync** (`cloudLoadAll` em `gdp-core.js`) — alvo da 20.15b. **Fora do escopo da 20.17.**
- **Realtime push** (`gdp-realtime.js`, via Supabase `postgres_changes`) — **alvo da 20.17.**

### Fluxo realtime atual (`gdp-realtime.js:157-231`, `handleEntityChange`)

Para um evento `UPDATE` de um pedido:

1. **Filtro de tenant** (L162-166): descarta eventos de outro `empresa_id`. OK, manter.
2. **Dirty window** (L167-176): `isDirty = (Date.now() - getLastLocalSave(lsKey)) < 5000`.
3. Se **dirty** (L188-205): só sobrescreve o item local se `remoteTs > localTs` (timestamp). Senão, protege o local.
4. Se **não-dirty** (L206-217): **substituição wholesale cega** — `items[j] = record`, sem olhar timestamp.

### A cadeia de save (`gdp-core.js:1389-1399`, `savePedidos`)

```js
function savePedidos() {
  pedidos = pedidos.map(sanitizePedidoLegacyData);
  saveWrappedArray(ORDERS_KEY, pedidos);
  _lastLocalSave[ORDERS_KEY] = Date.now();           // (A) reinicia a dirty window

    window.gdpApi.pedidos.save(p).catch(...);         //     cada um async, sem ordem garantida

```

E o `save()` genérico (`gdp-api.js:278-292`):
```js

  ...

```

## 2. Diagnóstico arquitetural (a causa-raiz REAL)

O handoff descreve "save em voo sobrescreve edição manual". Indo mais fundo, há **três defeitos compostos**:

### Defeito 1 — `savePedidos()` re-salva os 145 pedidos a cada chamada (`gdp-core.js:1396`)
Qualquer mudança em **um** pedido dispara 145 upserts assíncronos. Isso:
- multiplica por 145 a janela de corrida e o tráfego de eco;
- faz cada pedido inalterado receber um `updated_at` novo a cada operação não relacionada;
- é a fonte primária dos "ecos em voo" que chegam atrasados.

### Defeito 2 — O eco do próprio cliente não é distinguível do update de outro usuário
O `postgres_changes` devolve ao **próprio** cliente os upserts que ele fez (eco). O handler trata eco e mudança-remota-legítima de forma idêntica — só o `updated_at` arbitra. Como o eco carrega o `updated_at` setado em (C) no momento do upsert, ele pode ser **≥** ao `updated_at` da edição manual seguinte (especialmente sob latência de rede + os 145 saves em voo). Resultado: o eco vence e reverte.

### Defeito 3 — `updated_at` é gerado no cliente, no momento do upsert (não da edição)
Dois saves do mesmo pedido geram dois `updated_at` cliente-relativos. A ordem de chegada dos ecos **não é garantida**. Last-write-wins por timestamp-de-upsert é frágil quando o mesmo registro é salvo em rajada.

### Por que a dirty window de 5s não basta
Ela protege por **chave** (`gdp.pedidos.v1`), não por **registro**. Como todo save reinicia a janela global e re-salva tudo, a janela está quase sempre "suja" durante uso ativo — e mesmo dentro dela, a regra `remoteTs > localTs` deixa o eco passar quando os timestamps empatam ou o eco é marginalmente mais novo.

---

## 3. Princípio de design (decisão arquitetural central)

> **O eco do próprio cliente nunca deve sobrescrever o estado local. A arbitragem por timestamp continua válida apenas para mudanças genuinamente remotas (de outro cliente/aba).**

Três camadas, em ordem de prioridade:

1. **Echo suppression determinística** (resolve AC1/AC3 sem heurística de tempo): o cliente registra os `id`s que ele próprio acabou de upsertar; quando o `postgres_changes` devolve esse `id` com um payload idêntico ao que enviou, o handler **ignora** (é eco).
2. **Save seletivo** (reduz a superfície da corrida): `savePedidos(changedId)` persiste **apenas o pedido alterado**, não os 145. Mantém retrocompat (sem arg = salva todos, p.ex. no boot).
3. **Arbitragem residual por timestamp** (preserva AC4): para um `UPDATE` que **não** é eco do próprio cliente, mantém-se a regra atual (aplica se `remoteTs >= localTs`), garantindo que a mudança de outro usuário ainda chega.

Isso é o análogo realtime do `origin: user|system` da 20.15b: aqui a "origem" é **própria (eco) vs alheia (remota legítima)**.

## 4. Especificação para o @dev (por arquivo)

### 4.1 `gdp-api.js` — marcar a origem do upsert (registrar eco)

No `save()` genérico (L278), após o upsert bem-sucedido, registrar o `id` + um hash/sentinela do payload num buffer de "ecos esperados" com TTL curto. Expor um helper consultável pelo realtime.

```js
// novo, escopo do módulo gdp-api.js
var _selfEchoes = {};                 // id -> { sig, ts }
function _markSelfEcho(table, id, row) {
  _selfEchoes[table + ':' + id] = { sig: _rowSignature(row), ts: Date.now() };
}
window.gdpApiIsSelfEcho = function (table, id, record) {
  var k = table + ':' + id;
  var e = _selfEchoes[k];
  if (!e) return false;
  if (Date.now() - e.ts > 8000) { delete _selfEchoes[k]; return false; }  // TTL 8s
  // eco confirmado se a assinatura bate (mesmo updated_at/status que enviamos)
  return e.sig === _rowSignature(record);
};
```
- `_rowSignature(row)` = algo barato e estável: `row.updated_at + '|' + row.status` (suficiente para pedidos; o `updated_at` é único por upsert).
- Chamar `_markSelfEcho(table, item.id, row)` dentro de `save()` logo após setar `item.updated_at` (L281) e mapear a `row`.
- **Importante:** o sinal de eco usa o **mesmo `updated_at`** que vai pro banco — então o eco devolvido bate exato; um UPDATE de outro cliente terá `updated_at` diferente e **não** será suprimido.

### 4.2 `gdp-realtime.js` — suprimir eco no `handleEntityChange` (UPDATE)

No ramo `UPDATE` (L187), **antes** de qualquer escrita (tanto no caminho dirty quanto no não-dirty):

```js
} else if (type === 'UPDATE') {
  // Story 20.17: ignorar eco do próprio cliente (não sobrescreve edição local).
  if (typeof window.gdpApiIsSelfEcho === 'function' &&
      window.gdpApiIsSelfEcho(table, record.id, record)) {
    return false;  // é o nosso próprio upsert voltando — nada a fazer
  }
  // ... lógica atual (dirty window + timestamp) permanece para updates de OUTROS
}
```

- Para updates genuinamente remotos, manter a lógica existente **inalterada** (AC4/AC5 preservados).
- Recomendação adicional (defensiva, opcional): no caminho **não-dirty** (L206-217), trocar a substituição wholesale cega por uma condicionada a timestamp (`remoteTs >= localTs`), para alinhar com o caminho dirty e evitar reversões mesmo fora da janela. Baixo risco, melhora AC5.

### 4.3 `gdp-core.js` — save seletivo (`savePedidos`)

```js
function savePedidos(changedId) {            // arg opcional, retrocompat
  pedidos = pedidos.map(sanitizePedidoLegacyData);
  saveWrappedArray(ORDERS_KEY, pedidos);
  _lastLocalSave[ORDERS_KEY] = Date.now();
  syncPedidosGDPToEstoqueIntel(true);
  if (window.gdpApi && window.gdpApi.pedidos) {
    var alvo = changedId
      ? pedidos.filter(function (p) { return p.id === changedId; })
      : pedidos;                              // sem arg = comportamento atual (todos)
    alvo.forEach(function (p) {
      window.gdpApi.pedidos.save(p).catch(function (e) { gdpWarn('[Pedidos] save failed:', p.id, e.message); });
    });
  }
}
```
- Chamadas de status passam o id: `savePedidos(pedidoId)`.
- Pontos que mudam vários pedidos (bulk) podem chamar `savePedidos()` sem arg, ou iterar — decisão de implementação; o importante é não regredir o boot.

> **AUDITORIA (feita pelo @architect):** existem **38 chamadas** de `savePedidos()` no código. O arg opcional garante que as **36** que ficam sem arg mantêm o comportamento atual (salva todos). **Só migrar para seletivo** os handlers onde a corrida importa: `alterarStatusPedido` (gdp-pedidos.js:3204) e o faturar (gdp-notas-fiscais.js:1126). NÃO refatorar as 38 — risco desnecessário. Echo suppression (4.1/4.2) é a defesa principal; o save seletivo é otimização complementar.

### 4.4 `gdp-pedidos.js` + `gdp-notas-fiscais.js` — status + navegação de aba

- `alterarStatusPedido` (L3199): `savePedidos(pedidoId)` (seletivo).
- `aplicarSituacaoBulk` (L737): manter `savePedidos()` (vários) — ou iterar seletivo.
- **Navegação ao faturar** (`gdp-notas-fiscais.js:1125`, após `pedido.status = "faturado"`):
  ```js
  pedido.status = "faturado";
  savePedidos(pedido.id);
  if (typeof setPedidoStatusTab === 'function') setPedidoStatusTab('faturado'); // AC2
  renderAll();
  ```
  `setPedidoStatusTab('faturado')` (existe em `gdp-pedidos.js:3055`) ajusta `pedidoStatusTabAtual` e re-renderiza — o pedido aparece na aba certa em vez de "sumir".

## 5. Acceptance Criteria → cobertura do design

| AC | Como o design cobre |
|----|---------------------|
| AC1 — status manual não reverte | Echo suppression (4.1+4.2): o eco do próprio save é ignorado → nada sobrescreve a edição. |
| AC2 — faturar navega p/ aba Faturado | `setPedidoStatusTab('faturado')` após faturar (4.4). |
| AC3 — edição logo após faturar persiste | Save seletivo (4.3) reduz ecos em voo + echo suppression cobre o resto. |
| AC4 — update de OUTRO usuário aplica | Updates remotos têm `updated_at` ≠ do nosso eco → não suprimidos; lógica timestamp intacta. |
| AC5 — sem perda de edições concorrentes | Save seletivo + (opcional) condicionar o caminho não-dirty ao timestamp. |
| AC6 — bump de versão | Bump dos JS alterados no gdp-contratos.html. |

## 6. Ordem de implementação (commits isolados)

1. **`gdp-api.js`**: buffer de ecos + `window.gdpApiIsSelfEcho` + `_markSelfEcho` no `save()`. (núcleo)
2. **`gdp-realtime.js`**: guard de echo suppression no UPDATE. (núcleo)
3. **`gdp-core.js`**: `savePedidos(changedId)` seletivo, retrocompat.
4. **`gdp-pedidos.js`**: `alterarStatusPedido`/`aplicarSituacaoBulk` passam id.
5. **`gdp-notas-fiscais.js`**: `setPedidoStatusTab('faturado')` ao faturar.
6. **`gdp-contratos.html`**: bump `?v=`.

Cada passo é testável isoladamente; 1+2 já resolvem AC1/AC3 sozinhos.

## 7. Riscos e mitigações

| Risco | Mitigação |
|-------|-----------|
| Echo suppression ampla demais → perde update remoto legítimo | A assinatura inclui `updated_at` exato do nosso upsert; só o **nosso** eco bate. TTL de 8s evita supressão tardia. |
| Save seletivo quebrar fluxos que dependiam de salvar tudo | Arg opcional: sem arg = comportamento atual. Auditar chamadas de `savePedidos()` antes de tornar seletivas. |
| `setPedidoStatusTab` num contexto sem a aba renderizada (ex.: portal) | Guard `typeof setPedidoStatusTab === 'function'` antes de chamar. |
| `_rowSignature` fraca (colisão) | Usar `updated_at` (ISO ms, único por upsert) garante unicidade prática. |

## 9. Decisão arquitetural (resumo)

- **Determinístico > heurístico:** echo suppression por assinatura resolve a corrida na raiz, sem depender de afinar janelas de tempo.
- **Reduzir a superfície:** save seletivo ataca o agravante (145 upserts por clique).
- **Preservar o que funciona:** a arbitragem por timestamp para updates remotos fica intacta — AC4/AC5 não regridem.
- **Consistência com 20.15b:** mesma filosofia de "origem do dado decide a arbitragem".
