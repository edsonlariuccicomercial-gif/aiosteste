# Story 7.3 — Corrigir Race Condition NF-e com Funcao Atomica

**Epic:** Epic 7 — Modernizacao e Remediacao de Divida Tecnica
**Status:** Ready
**Priority:** P0 (CRITICAL — Sprint 0)
**Story Points:** 3

---

## Descricao

O sistema atual gera numeros de NF-e atraves de um fluxo **nao-atomico**: o codigo em `gdp-api.js` faz um SELECT no `nf_counter` para obter o numero atual, incrementa no JavaScript, e entao faz um UPDATE separado. Se dois usuarios (ou duas abas do mesmo usuario) emitirem NF-e simultaneamente, ambos lerao o mesmo numero, gerando **NF-e duplicada**.

### Problema de Negocio

- NF-e com numero duplicado e **rejeitada pela SEFAZ**
- Rejeicao impede faturamento ate correcao manual
- Multa fiscal por irregularidade
- Com apenas 2 usuarios simultaneos, a duplicacao e **inevitavel**
- Alem disso, o indice em `notas_fiscais(numero)` nao e UNIQUE por empresa/serie, permitindo que duplicatas sejam salvas no banco sem erro

### Solucao

1. Criar funcao PostgreSQL `next_nf_number()` com `SELECT ... FOR UPDATE` que garante atomicidade
2. Substituir o fluxo JS (GET + UPDATE) por uma chamada RPC unica
3. Criar indice UNIQUE em `notas_fiscais(empresa_id, numero, serie)` como safety net

## Notas Tecnicas

### Script SQL Pronto (Migrations 007 + 008)

**Migration 007 — Funcao atomica** (referencia: `db-specialist-review.md` Script 2):

```sql
CREATE OR REPLACE FUNCTION next_nf_number(p_empresa_id TEXT)
RETURNS INTEGER
LANGUAGE plpgsql
AS $$
DECLARE
  v_next INTEGER;
BEGIN
  SELECT counter + 1 INTO v_next
  FROM nf_counter
  WHERE empresa_id = p_empresa_id
  FOR UPDATE;

  IF v_next IS NULL THEN
    INSERT INTO nf_counter (empresa_id, counter, updated_at)
    VALUES (p_empresa_id, 1, now())
    ON CONFLICT (empresa_id) DO UPDATE
      SET counter = nf_counter.counter + 1, updated_at = now()
    RETURNING counter INTO v_next;
  ELSE
    UPDATE nf_counter
    SET counter = v_next, updated_at = now()
    WHERE empresa_id = p_empresa_id;
  END IF;

  RETURN v_next;
END;
$$;
```

**Migration 008 — UNIQUE index + fix trigger** (referencia: `db-specialist-review.md` Script 3):

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_nfs_unique_empresa_numero_serie
  ON notas_fiscais(empresa_id, numero, serie);

CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS trg_resultados_updated_at ON resultados_orcamento;
CREATE TRIGGER trg_resultados_updated_at
  BEFORE UPDATE ON resultados_orcamento
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

### Alteracao no Frontend (gdp-api.js)

**Antes (vulneravel):**
```javascript
// Logica atual: GET + incremento JS + save
const counter = await nfCounterApi.get(empresaId);
const nextNumber = counter.value + 1;
await nfCounterApi.save(empresaId, nextNumber);
```

**Depois (atomico):**
```javascript
// Nova logica: chamada RPC unica
const { data: nextNumber, error } = await supabase.rpc('next_nf_number', {
  p_empresa_id: empresaId
});
if (error) throw new Error(`Falha ao gerar numero NF: ${error.message}`);
```

### Verificacao Pre-Deploy

Antes de criar o UNIQUE index, verificar se ja existem duplicatas:

```sql
SELECT empresa_id, numero, serie, COUNT(*)
FROM notas_fiscais
GROUP BY empresa_id, numero, serie
HAVING COUNT(*) > 1;
```

Se houver duplicatas, resolver manualmente antes de aplicar a migration.

| Risco | Probabilidade | Mitigacao |
|-------|:-------------:|-----------|
| NFs duplicadas ja existem no banco (UNIQUE index falha) | MEDIA | Verificar com query acima ANTES de aplicar migration |
