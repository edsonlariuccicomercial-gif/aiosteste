# Story 14.3: Paginação API + Lazy Loading Frontend

## Status: Ready for Review
## Epic: EPIC-14-SYNC-PERFORMANCE
## Priority: P1-High
## Agent: @dev (Dex)
## Blocked By: 14.1

## Problema

O sistema carrega TODOS os dados em memória no boot (`gdp-core.js:720-759`) e renderiza tabelas inteiras sem paginação (`gdp-pedidos.js:49-67`). Em máquinas com datasets grandes ou conexão lenta, isso causa lentidão severa.

## Acceptance Criteria

- [ ] AC1: `gdp-api.js .list()` aceita parâmetros `limit` e `offset` (default: limit=50)
- [ ] AC2: Tabela de pedidos renderiza com paginação (50 por página, navegação prev/next)
- [ ] AC3: Tabela de contratos renderiza com paginação (50 por página)
- [ ] AC4: Boot inicial carrega apenas página 1 de cada entidade (não todo o dataset)
- [ ] AC5: `cloudLoadAll()` em `app-sync.js:63-82` usa `Promise.all()` para candidatos (não sequencial)
- [ ] AC6: Tempo de carregamento inicial < 3 segundos com 500+ registros no Supabase
- [ ] AC7: Busca/filtro funciona com dados paginados (server-side quando possível)

## Notas Técnicas

- Supabase suporta `range` header nativamente: `.select('*').range(0, 49)` retorna 50 rows
- Para contagem total: `.select('*', { count: 'exact', head: true })` retorna count sem dados
- Busca server-side: `.ilike('nome', '%termo%')` — mais eficiente que filtro em memória
- `Promise.all()` no cloudLoadAll reduz latência de N requests sequenciais para 1 round-trip
