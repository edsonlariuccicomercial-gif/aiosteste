# Story 20.3 — Contas a Receber: corrigir data de emissão vazia no modal

**Epic:** EPIC-20 — Ajustes Financeiro GDP
**Status:** Ready for Review
**Prioridade:** P1
**Risco:** BAIXO | **Complexidade:** S
**Onda:** 1 (quick-win, sem dependências)
**Executor:** @dev
**Resolve:** FIN-3 — ao abrir os detalhes de uma conta a receber, a "Data de Emissão" vem vazia.

## Story
**Como** gestor financeiro,
**Quero** que a data de emissão apareça preenchida ao abrir os detalhes de uma conta a receber,
**Para** não ver o campo vazio em contas antigas ou geradas por caminhos diferentes.

## Contexto / Problema
O modal lê `conta.dataEmissao` (`gdp-init.js:428` display / `:460` edit / `:486` save) — o nome está correto. Mas registros legados ou criados por outro caminho podem não ter o campo, ou pode haver divergência snake_case (`data_emissao` vindo do Supabase) vs camelCase (`dataEmissao`). `buildReceivableFromInvoice` popula corretamente (`gdp-notas-fiscais.js:840`), então o problema é nos dados que não passaram por esse caminho.

**Decisão do stakeholder (Edson, 2026-06-12):** corrigir via **fallback na exibição** — NÃO rodar migration/backfill no banco.

## Critérios de Aceitação
- **AC1:** *Given* uma conta com `dataEmissao` ausente, *When* abro os detalhes, *Then* o campo exibe uma data válida (data da NF de origem ou data de criação do registro) e nunca aparece vazio.
- **AC2:** *Given* uma conta com `dataEmissao` presente, *When* abro os detalhes, *Then* o valor original é exibido sem alteração.
- **AC3 (não-regressão):** salvar/editar a conta não corrompe a data exibida.

## Escopo
- **IN:** normalização/fallback na leitura ao exibir e editar (`dataEmissao || data_emissao || data da NF de origem || data de criação`).
- **OUT:** migration/backfill no Supabase; alteração de dados persistidos.

## Requisitos Funcionais
- **FR-20.3.1:** Ao exibir/editar, ler `conta.dataEmissao || conta.data_emissao` e, se ausente, fazer fallback para a data de emissão da NF de origem (via `notaFiscalId`/`origemId`) ou data de criação do registro.
- **FR-20.3.2:** O fallback ocorre apenas na camada de exibição; nada é gravado no banco automaticamente por esta story.
