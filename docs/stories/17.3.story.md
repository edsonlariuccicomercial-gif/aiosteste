# Story 17.3 — Fonte Única do Caixa + Backfill do Supabase

**Epic:** EPIC-17 — Sincronização do Caixa Multiusuário
**Status:** Ready for Review
**Prioridade:** P1
**Risco:** MÉDIO | **Complexidade:** M
**Executor:** @dev
**Resolve:** Causa B (S3 — visível mas não conta)

## Story
**Como** usuário do caixa,
**Quero** que todos os lançamentos visíveis contabilizem no saldo, e que um navegador novo carregue os dados do Supabase,
**Para** que o saldo seja sempre correto e consistente.

## Contexto / Problema
`getCaixaResumo` (gdp-pedidos.js L2023-2038) usa fonte dupla: `loadConciliacao()` OU fallback legado `caixaExtratoMovimentos`. `loadConciliacao` só lê localStorage, sem backfill. Resultado: navegador sem dados cai no fallback → 161 visíveis mas saldo = só inicial.

## Critérios de Aceitação
- **AC1:** *Given* navegador novo sem cache, *When* abre o caixa, *Then* lançamentos vêm do Supabase (backfill) e o saldo bate.
- **AC2:** *Given* lançamentos visíveis, *When* o saldo é calculado, *Then* TODOS contabilizam (visível = contado).
- **AC3:** Não há mais cenário "161 visíveis, saldo = só inicial".

## Requisitos Funcionais
- **FR-17.3.1:** Eliminar (ou neutralizar) o fallback `caixaExtratoMovimentos` em `getCaixaResumo`; render e cálculo leem a mesma fonte.
- **FR-17.3.2:** `loadConciliacao` faz backfill do Supabase no primeiro load.
- **FR-17.3.3:** Garantir visível = contabilizado.

## Escopo
- **IN:** `getCaixaResumo` fonte única, `loadConciliacao` backfill.
- **OUT:** empresa_id (17.1/17.2), saldo inicial (17.4).

## Dev Notes (Dex)
**Pré-passo (decisão do stakeholder):** revertidos os 5 arquivos de código do EPIC-16 ao estado pré-EPIC-16 (`git checkout 70f8118 -- <arquivos>`), alinhando o código local com produção (`cxawxu19k`). Docs de story do EPIC-16 preservados (histórico). Implementei a 17.3 sobre essa base limpa.

**Refino do diagnóstico durante a implementação:** descobri que o backfill do Supabase para conciliação **JÁ EXISTE e JÁ RODA no boot** — `gdp.conciliacao.v1`→`conciliacoes` está no `_tableMap` (gdp-init.js L2895), e `renderAll`→`_renderAllImmediate` já chama `renderCaixa()` quando a aba ativa é o caixa. Então o item 1 (backfill) NÃO precisou de código novo.

**A causa real do Edson** era a combinação: (a) `getCaixaResumo` usava o **fallback legado `caixaExtratoMovimentos`** quando `loadConciliacao()` parecia vazio/defasado, e (b) esse fallback é uma fonte em memória divergente do banco. Com o fallback removido, o cálculo passa a ler SEMPRE do espelho do Supabase (já merged no boot).

**Itens implementados:**
- **Item 2 (FONTE ÚNICA) — gdp-pedidos.js `getCaixaResumo`:** removido o fallback `caixaExtratoMovimentos`; `items = concItems` (sempre `loadConciliacao`). Render e cálculo na mesma fonte → "visível = contabilizado".
- **Item 3 (DEFENSIVO) — gdp-realtime.js `handleEntityChange`:** descarta eventos com `empresa_id` != `getEmpresaId()` (cross-tenant). Hoje inócuo (tudo LARIUCCI) mas previne regressão futura. Reaproveita a lógica que já existia em `isOurRecord`.
- **Item 1 (BACKFILL):** já existente no boot — sem código novo (documentado).

**Não-regressão:** `caixaExtratoMovimentos` NÃO foi removida — segue usada por `sincronizarExtratoCaixaViaApi`/`conciliarCaixaViaApi` (L2362-2391) e pelo contador (L1628). Só saiu do cálculo do saldo. Aba de conciliação por extrato intacta. `node --check` OK; vitest 183/183, 0 falhas.

**Escopo do EPIC-17:** 17.1 (empresa_id código) e 17.2 (migração) ficaram DESNECESSÁRIAS (banco já unificado). 17.3 resolve a causa real.
