# Story 7.4 — Implementar Backup/DR para Dados Fiscais

**Epic:** Epic 7 — Modernizacao e Remediacao de Divida Tecnica
**Status:** Ready
**Priority:** P0 (CRITICAL — Sprint 0)
**Story Points:** 5

---

## Descricao

O sistema armazena dados fiscais com **obrigacao legal de guarda por 5 anos** (Art. 174 do Codigo Tributario Nacional) em um Supabase Free Tier que nao oferece SLA, nao possui backup externo configurado e pode ter dados removidos a qualquer momento por inatividade ou estouro de limites.

Dados em risco:
- NF-e autorizadas (XMLs + metadados)
- Certificado digital PFX
- Historico de emissoes
- Contas a receber/pagar vinculadas a NF-e
- Audit log de operacoes fiscais

### Problema de Negocio

- Perda de dados fiscais e **irreversivel** — nao ha como reconstruir NF-e autorizadas
- Descumprimento do Art. 174 CTN: guarda obrigatoria de documentos fiscais por 5 anos
- Supabase Free Tier: 500MB de DB, sem garantia de disponibilidade, sem backup automatico
- Se o banco corromper ou a conta for suspensa, a empresa perde TODAS as NF-e emitidas

### Solucao

Implementar export diario automatico dos dados fiscais criticos para storage externo (Supabase Storage bucket ou alternativa), com RPO de 24h e capacidade de restauracao.

---

## Criterios de Aceitacao

### AC-1: Export diario automatico configurado
- [ ] Funcao serverless (ou cron job) executa diariamente a exportacao de dados fiscais
- [ ] Exporta: `notas_fiscais`, `nf_counter`, `contas_receber` (vinculadas a NF), `audit_log` (operacoes fiscais)
- [ ] Formato: JSON compactado (gzip) com timestamp no nome do arquivo
- [ ] Execucao automatica via Vercel Cron ou similar (schedule: `0 3 * * *` — 3h da manha)

### AC-2: Storage externo funcional
- [ ] Bucket configurado (opcoes: Supabase Storage, AWS S3, Google Cloud Storage, ou Backblaze B2)
- [ ] Credenciais armazenadas em env vars do servidor (nunca no frontend)
- [ ] Bucket com politica de retencao de 5 anos (ou configuracao manual equivalente)
- [ ] Verificacao de que o upload foi bem-sucedido (checksum ou confirmacao de storage)

### AC-3: Exportacao de XMLs de NF-e
- [ ] XMLs autorizados (campo `xml_autorizado` em `notas_fiscais`) exportados individualmente ou em batch
- [ ] Nomenclatura padrao: `{empresa_id}/{ano}/{mes}/NFe-{numero}-{serie}-{chave_acesso}.xml`
- [ ] Dados mantidos por no minimo 5 anos (regra de retencao configurada)

### AC-4: Monitoramento e alertas
- [ ] Log de cada execucao de backup (sucesso/falha)
- [ ] Se backup falhar, alerta enviado (console.error no minimo; email/webhook desejavel)
- [ ] Dashboard ou endpoint que mostra: ultimo backup bem-sucedido, quantidade de registros exportados, tamanho total

### AC-5: Capacidade de restauracao
- [ ] Documentacao de como restaurar dados do backup para o banco
- [ ] Script de restauracao testado pelo menos 1 vez
- [ ] RPO (Recovery Point Objective): maximo 24h de perda de dados
- [ ] RTO (Recovery Time Objective): maximo 4h para restaurar

### AC-6: Certificado digital com backup seguro
- [ ] Certificado PFX (campo `config_fiscal` em `empresas`) incluido no backup
- [ ] Backup do certificado CRIPTOGRAFADO (nao em plaintext)
- [ ] Senha do certificado armazenada separadamente das credenciais de backup

## Notas Tecnicas

### Arquitetura Recomendada

```
[Vercel Cron - 3h diario]
       |
       v
[Serverless Function: backup-fiscal.js]
       |
       ├── Query: notas_fiscais WHERE updated_at > last_backup
       ├── Query: nf_counter (sempre completo, tabela pequena)
       ├── Query: contas_receber WHERE origem_tipo = 'nota_fiscal'
       ├── Query: audit_log WHERE tabela IN ('notas_fiscais', 'nf_counter')
       |
       v
[Compactar em JSON.gz com timestamp]
       |
       v
[Upload para Supabase Storage / S3 / Backblaze B2]
       |
       v
[Log resultado no audit_log]
```

### Opcoes de Storage (comparativo)

| Opcao | Custo/GB/mes | Retencao | Complexidade | Recomendacao |
|-------|:------------:|:--------:|:------------:|:------------:|
| Supabase Storage (mesmo projeto) | Free (1GB) | Manual | Baixa | MVP — comeca aqui |
| Backblaze B2 | $0.005 | Configuravel | Media | Melhor custo-beneficio |
| AWS S3 Glacier | $0.004 | Lifecycle rules | Alta | Overkill para volume atual |
| Google Cloud Storage | $0.02 | Lifecycle rules | Media | Alternativa |

**Recomendacao:** Iniciar com Supabase Storage (mesmo ecossistema, sem custo adicional, 1GB free). Se volume crescer, migrar para Backblaze B2.

### Script de Export (pseudocodigo)

```javascript
// api/backup-fiscal.js
import { createClient } from '@supabase/supabase-js';
import { gzip } from 'zlib';

export default async function handler(req, res) {
  const supabase = createClient(URL, SERVICE_ROLE_KEY);
  
  // 1. Buscar dados desde ultimo backup
  const lastBackup = await getLastBackupTimestamp();
  
  const { data: nfs } = await supabase
    .from('notas_fiscais')
    .select('*')
    .gt('updated_at', lastBackup);
  
  const { data: counter } = await supabase
    .from('nf_counter')
    .select('*');
  
  // 2. Compactar
  const payload = JSON.stringify({ nfs, counter, timestamp: new Date() });
  const compressed = gzipSync(Buffer.from(payload));
  
  // 3. Upload para storage
  const filename = `backups/fiscal/${new Date().toISOString().split('T')[0]}.json.gz`;
  await supabase.storage
    .from('fiscal-backups')
    .upload(filename, compressed);
  
  // 4. Log
  await supabase.from('audit_log').insert({
    empresa_id: 'SYSTEM',
    action: 'BACKUP_FISCAL',
    details: { records: nfs.length, size: compressed.length }
  });
  
  return res.json({ success: true, records: nfs.length });
}
```

### Configuracao do Cron (Vercel)

No `vercel.json`:
```json
{
  "crons": [
    {
      "path": "/api/backup-fiscal",
      "schedule": "0 3 * * *"
    }
  ]
}
```

**NOTA:** Vercel Crons requer plano Pro ($20/mes) para crons diarios. Alternativa no Free Tier: usar servico externo (cron-job.org, GitHub Actions scheduled) para chamar o endpoint.

## Riscos

| Risco | Probabilidade | Mitigacao |
|-------|:-------------:|-----------|
| Vercel Hobby nao suporta cron nativo | ALTA | Usar servico externo (cron-job.org) para trigger HTTP |
| Storage atinge 1GB (Supabase free) | MEDIA | Monitorar volume; migrar para Backblaze B2 se necessario |
| Backup falha silenciosamente | MEDIA | Log obrigatorio + alerta se > 24h sem backup |
| Export de XMLs grandes causa timeout (30s Vercel) | BAIXA | Paginar em batches de 100 NFs; usar streaming |
