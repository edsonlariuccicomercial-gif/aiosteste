## Problema

O usuário pediu para que as buscas do sistema funcionem **sem depender de digitar acentuação e pontuação** (ex.: buscar "feijao" deve achar "Feijão"; "12345678" deve achar "12.345.678"). Foi feita uma tentativa anterior (Story 20.10, helper `normalizeSearch`), mas a busca **continua exigindo acento** em vários pontos.

## Causa Raiz (evidência de código — @analyst)

São **duas causas independentes**:

### Causa A — Regex de normalização frágil/corrompível
O helper global `window.normalizeSearch` em `gdp-core.js:28-36` remove diacríticos com um regex que usa **caracteres literais combining** (`/[<combining-marks>]/g`) em vez do escape Unicode `/[̀-ͯ]/g`. Caracteres combining literais no código-fonte são frágeis: dependendo do encoding/salvamento do arquivo, podem corromper e o regex deixa de remover acentos. A mesma variante suspeita existe em `gdp-conta-corrente.js:30`. Outros arquivos do projeto já usam o escape correto (`̀-ͯ`) — daí a inconsistência.

### Causa B — 5 buscas nunca migradas para `normalizeSearch`
Cinco call-sites de busca/filtro ainda usam apenas `.toLowerCase()` (sem remoção de acento/pontuação), ignorando o helper:

| Arquivo:linha | Função | Campos buscados |
|---------------|--------|-----------------|
| `gdp-banco-produtos.js:116-118` | `renderBancoProdutos()` | descricao, sku |
| `gdp-dashboard.html:612` | `renderSchools()` | escola.nome |
| `gdp-dashboard.html:681` | `filterCatalog()` (inline) | produto, escola |
| `gdp-estoque-intel-mobile.html:495` | `renderMobileStock()` | produto.nome |
| `gdp-portal.html:1325` | `filterCatalog()` | produto name |

## Princípio da Correção

1. Corrigir o regex de `normalizeSearch` para usar o escape Unicode `̀-ͯ` (robusto a encoding), aplicando em `gdp-core.js` e `gdp-conta-corrente.js`.
2. Migrar os 5 call-sites para `window.normalizeSearch` **em ambos os lados** da comparação (termo digitado E campo do registro).
