# Story 14.1: Fix identidade sync + empresa_id consistente

## Status: Ready for Review
## Epic: EPIC-14-SYNC-PERFORMANCE
## Priority: P0-Critical
## Agent: @dev (Dex)

## Problema

`getSyncUserId()` e `getEmpresaId()` derivam a identidade da empresa do `localStorage`, que varia entre máquinas. Resultado: dados salvos em uma máquina ficam invisíveis em outra.

## Acceptance Criteria

- [ ] AC1: Login do portal (`gdp-portal.html`) e dashboard define `empresa_id` a partir do `escolas-credentials.json`, NÃO do localStorage arbitrário
- [ ] AC2: `getSyncUserId()` em `app-sync.js` usa `empresa_id` fixo derivado das credenciais autenticadas
- [ ] AC3: `getEmpresaId()` em `gdp-api.js` usa a mesma fonte de verdade que `getSyncUserId()`
- [ ] AC4: Ao autenticar escola (ex: `login=lauriston`), o sistema seta `localStorage["nexedu.empresa"].syncUserId` = `id` do registro em `escolas-credentials.json` (ex: `"lauriston-souza"`)
- [ ] AC5: Ao autenticar admin (gestor), `empresa_id` = `"LARIUCCI"` (empresa gestora)
- [ ] AC6: Dados existentes no Supabase com `empresa_id` antigo continuam acessíveis (migração ou alias)
- [ ] AC7: Testes manuais confirmam: mesma escola em 2 máquinas vê mesmos dados

## Tasks

- [x] T1: Portal doLogin() seta `syncUserId = escola.id` no localStorage ao autenticar
- [x] T2: getSyncUserId() em app-sync.js já prioriza syncUserId (sem mudança necessária na lógica, fix está no login)
- [x] T3: getEmpresaId() em gdp-api.js já prioriza syncUserId (mesma lógica, fix está no login)
- [x] T4: Dashboard login.html seta `syncUserId = "LARIUCCI"` para admin (legacy + Supabase Auth)
- [ ] T5: Migrar dados existentes no Supabase: UPDATE rows com empresa_id antigo para novo id padronizado
- [ ] T6: Testar login escola + login admin em máquinas diferentes

## Arquivos Afetados

| Arquivo | Alteração |
|---------|-----------|
| `dashboard/gdp-portal.html` | doLogin() seta `syncUserId = escola.id` no localStorage |
| `dashboard/login.html` | Admin login seta `syncUserId = "LARIUCCI"` (legacy + Supabase Auth) |
| `dashboard/app-sync.js` | Comentário documentando fix (lógica já priorizava syncUserId) |
| `dashboard/gdp-api.js` | Comentário documentando fix (lógica já priorizava syncUserId) |

## Notas Técnicas

- `escolas-credentials.json` já tem campo `id` único por escola (ex: `"alceu-novaes"`, `"lauriston-souza"`)
- A função `resolveEmpresaId()` deve ser determinística: mesmo login → mesmo id, sempre
- Fallback chain atual (`syncUserId > nomeFantasia > nome > cnpj > "default"`) deve ser substituída por id fixo

## Definition of Done

- [ ] Código implementado e testado
- [ ] Dados existentes migrados ou com alias funcional
- [ ] Duas máquinas com mesmo login veem mesmos dados
- [ ] Nenhuma regressão no fluxo de login admin/gestor
