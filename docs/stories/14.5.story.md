# Story 14.5: Numeração NF-e — respeitar proximoNumero configurado pelo usuário

## Status: Ready for Review
## Epic: EPIC-14-SYNC-PERFORMANCE
## Priority: P0-Critical
## Agent: @dev (Dex)

## Problema

O campo "Próximo número" nas configurações gerais de nota fiscal (`nexedu.config.notas-fiscais.proximoNumero`) é salvo corretamente na UI, mas **nunca é lido** pelo fluxo de emissão. A função `getProximoNumeroNf()` em `gdp-notas-fiscais.js:473-505` busca o número exclusivamente do Supabase RPC (`next_nf_number`) ou de um fallback local, ignorando completamente a configuração do usuário.

Resultado: o usuário configura o número 150, mas o sistema emite com o número que vier do banco (ex: 1209), causando notas com numeração errada.

## Acceptance Criteria

- [ ] AC1: Quando o usuário configura `proximoNumero` nas configurações gerais de NF-e, esse número DEVE ser usado na próxima emissão
- [ ] AC2: Após emissão, o `proximoNumero` na config auto-incrementa para o próximo número (ex: 150 → 151)
- [ ] AC3: O `nf_counter` no Supabase é sincronizado com o número usado (para manter consistência multi-máquina)
- [ ] AC4: Se `proximoNumero` NÃO estiver configurado (vazio ou zero), fallback para Supabase RPC `next_nf_number` (comportamento atual preservado)
- [ ] AC5: O campo `proximoNumero` na UI reflete o próximo número após salvar configurações E após cada emissão
- [ ] AC6: Nenhuma regressão no fluxo de emissão existente (SEFAZ homologação/produção)

## Tasks

- [x] T1: Refatorar `getProximoNumeroNf()` — lê `proximoNumero` da config como fonte primária
- [x] T2: Após uso, incrementa `proximoNumero` na config e salva no localStorage
- [x] T3: Sincroniza `nf_counter` no Supabase via `gdpApi.nf_counter.save()` (fire-and-forget)
- [x] T4: Fallback para Supabase RPC preservado quando `proximoNumero` vazio ou zero
- [ ] T5: Testar em produção: configurar número → emitir NF → verificar

## Arquivos Afetados

| Arquivo | Alteração |
|---------|-----------|
| `dashboard/js/gdp-notas-fiscais.js` | Refatorar `getProximoNumeroNf()` linhas 473-505 |
| `dashboard/app-config.js` | Sem mudança (já salva `proximoNumero` corretamente) |

## Notas Técnicas

- `nexedu.config.notas-fiscais` é o localStorage key da config de NF-e
- `proximoNumero` é o campo dentro da config (string numérica)
- `getProximoNumeroNf()` é async e retorna string
- Prioridade do fix: Config do usuário → Supabase RPC → fallback local
- `gdpApi.nf_counter.save()` faz upsert por `empresa_id` (não por `id`)

## Análise de Risco

- **Alto**: Numeração errada gera rejeição na SEFAZ e multa fiscal
- **Mitigação**: Testar em homologação antes de produção
- **Rollback**: Se falhar, reverter para comportamento atual (Supabase RPC only)

## Definition of Done

- [ ] Número configurado pelo usuário é usado na emissão
- [ ] Auto-incremento funciona após cada emissão
- [ ] Sincronização com Supabase nf_counter
- [ ] Fallback preservado quando não configurado
- [ ] Testado em cenário de emissão real (homologação)
