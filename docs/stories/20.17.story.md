## Problema

Dois sintomas relacionados no módulo de Pedidos:

1. **Ao faturar um pedido** (emitir NF-e), ele **não muda automaticamente para a aba de status correta** ("Faturado") — o usuário tem a percepção de que "sumiu".
2. **Ao alterar o status manualmente**, o pedido **volta sozinho para o status anterior** automaticamente, impedindo a correção do status.

## Causa Raiz (evidência de código — @analyst)

### Sintoma 1 — Não troca de aba ao faturar
`gerarNotaFiscalPedido()` em `gdp-notas-fiscais.js:1125` **seta corretamente** `pedido.status = "faturado"` e chama `savePedidos()` + `renderAll()`. Porém a aba ativa (`pedidoStatusTabAtual`) **não navega** — se o usuário está na aba "Em Aberto", o filtro de render (`gdp-pedidos.js:213`) exclui o pedido agora "faturado" e ele desaparece da visão atual, em vez de seguir o usuário.

### Sintoma 2 — Status manual reverte (race condition de realtime)
- `savePedidos()` (`gdp-core.js:1389-1398`) persiste **cada pedido async individualmente** no Supabase (`gdpApi.pedidos.save(p)`).
- `gdp-api.js:281` seta `item.updated_at = new Date().toISOString()` no momento do save.
- Quando o usuário edita o status manualmente, **saves ainda em voo** (de uma operação anterior) chegam de volta como eventos `postgres_changes` **depois** da edição, com `updated_at` mais novo.
- O handler de realtime `gdp-realtime.js:196-217` faz **substituição wholesale** do item local pelo registro remoto quando o remoto é "mais novo", **sobrescrevendo a edição manual**.
- A "dirty window" de 5s (`gdp-realtime.js:172-176`) não protege quando há **múltiplos saves em voo**.

## Princípio da Correção

1. **Reconciliação de realtime:** não sobrescrever uma edição local mais recente do usuário com um evento de save antigo em voo. A edição manual do usuário deve vencer eventos de eco do próprio cliente. (Estratégia a validar com @architect — ver Story 20.15b, que tem design correlato de origem `user` vs `system`.)
2. **UX de faturar:** ao faturar com sucesso, **navegar automaticamente para a aba "Faturado"** e renderizar o pedido lá (decisão do stakeholder).

## Acceptance Criteria (Given/When/Then)

### AC1 — Status manual NÃO reverte
```
DADO um pedido em qualquer status
QUANDO o usuário altera o status manualmente (alterarStatusPedido ou aplicarSituacaoBulk)
ENTÃO o pedido permanece no novo status
  E NÃO retorna automaticamente ao status anterior por eco de realtime
```

### AC2 — Faturar navega para a aba "Faturado"
```
DADO um pedido sendo faturado (emissão de NF-e com sucesso)
QUANDO o faturamento conclui
ENTÃO o status vira "faturado"
  E a aba ativa muda automaticamente para "Faturado"
  E o pedido aparece visível nessa aba (não "some")
```

### AC3 — Edição manual logo após faturar é respeitada
```
DADO um pedido recém-faturado
QUANDO o usuário corrige o status manualmente em seguida
ENTÃO a correção persiste e não é revertida por save em voo do faturamento
```

### AC4 — Realtime de OUTRO usuário ainda atualiza
```
DADO 2 usuários na mesma empresa
QUANDO o usuário B altera o status de um pedido
ENTÃO o usuário A vê a atualização (a proteção é só contra ECO do próprio cliente, não contra mudança legítima remota)
```

### AC5 — Sem perda de outras edições concorrentes
```
DADO saves de campos distintos do mesmo pedido em sequência
QUANDO os ecos de realtime chegam
ENTÃO nenhuma edição legítima do usuário é perdida (reconciliação por campo/timestamp, não wholesale cego)
```

### AC6 — Bump de versão dos JS alterados no HTML

## Escopo

**IN:**
- Reconciliação de realtime que não reverte edição manual local recente (`gdp-realtime.js`).
- Navegação automática para a aba "Faturado" ao faturar (`gdp-notas-fiscais.js` + `gdp-pedidos.js`).
- Garantir `updated_at`/marcação de origem coerente no fluxo de save (`gdp-core.js`/`gdp-api.js`).

**OUT:**
- Recalibração geral das regras de sync local↔nuvem (Story 20.15b).
- Fluxo de emissão NF-e em si (Story 20.14, já entregue).

## Riscos

- **MÉDIO.** Toca realtime/reconciliação em produção. Risco de regressão: bloquear updates legítimos de outros usuários (AC4) se a proteção contra eco for ampla demais.
- Recomendado @architect validar a estratégia (correlato à Story 20.15b — reutilizar conceito `origin: user|system`).
- Teste em 2 navegadores obrigatório (edição local não reverte; update remoto legítimo aplica).

## Teste

1. **AC1:** alterar status manual → aguardar > janela de eco → status permanece.
2. **AC2:** faturar pedido → aba muda para "Faturado" e pedido aparece.
3. **AC3:** faturar e logo corrigir status → correção persiste.
4. **AC4:** 2 navegadores — B muda status, A vê a mudança.
5. **AC5:** edições sequenciais de campos distintos → nenhuma se perde.
