# Story 7.1 — Implementar RLS em Todas as Tabelas

**Epic:** Epic 7 — Modernizacao e Remediacao de Divida Tecnica
**Status:** Ready
**Priority:** P0 (CRITICAL — Sprint 0)
**Story Points:** 5

---

## Descricao

Atualmente, 11 de 13 tabelas do sistema **nao possuem Row Level Security (RLS)**. Combinado com a chave `anon` do Supabase exposta no frontend, isso significa que qualquer pessoa com a URL pode executar CRUD irrestrito em todas as tabelas operacionais — incluindo leitura de contratos, precos, dados fiscais e ate o certificado digital PFX armazenado em `empresas.config_fiscal`.

Apenas `resultados_orcamento` e `preco_historico` possuem RLS ativo (migrations 004 e 005).

### Problema de Negocio

- Concorrentes podem ler contratos, precos e margens de lucro
- Certificado digital A1 (PFX + senha) acessivel = fraude fiscal possivel
- Dados de NF-e de todas as empresas expostos publicamente
- Viola LGPD para dados de fornecedores

### Solucao

Habilitar RLS em todas as 11 tabelas restantes e criar policies de isolamento por `empresa_id`, garantindo que cada empresa so acessa seus proprios dados. O script SQL esta pronto (migration 006) no `db-specialist-review.md`.

## Criterios de Aceitacao

### AC-1: Habilitar RLS em todas as tabelas operacionais
- [ ] `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` executado para: `empresas`, `clientes`, `contratos`, `pedidos`, `notas_fiscais`, `contas_receber`, `contas_pagar`, `entregas`, `nf_counter`, `data_snapshots`, `audit_log`
- [ ] Confirmar via query: `SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname = 'public'` — todas com `rowsecurity = true`

### AC-2: Policies de isolamento por empresa_id
- [ ] Policy `empresas_isolation` criada para tabela `empresas` usando `id = current_setting('app.current_empresa_id', true)`
- [ ] Policy generica `{tabela}_isolation` criada para todas as tabelas com `empresa_id`
- [ ] Policy `nf_counter_isolation` criada para `nf_counter` (PK = empresa_id)
- [ ] Todas as policies cobrem operacoes `FOR ALL` (SELECT, INSERT, UPDATE, DELETE)

### AC-3: Frontend configurado para enviar empresa_id via sessao
- [ ] Ao inicializar o Supabase client, configurar `app.current_empresa_id` via `supabase.rpc('set_config', ...)` ou header customizado
- [ ] Todas as queries do frontend continuam funcionando normalmente apos RLS ativo
- [ ] Testes manuais: usuario logado ve apenas dados da propria empresa

### AC-4: Service role mantido para backend
- [ ] Serverless functions (API) continuam usando `service_role` key (bypassa RLS)
- [ ] Confirmar que `service_role` NAO esta exposto no frontend (apenas em env vars do servidor)

### AC-5: Nenhuma regressao em fluxos existentes
- [ ] Fluxo de pedidos funciona (criar, editar, listar)
- [ ] Fluxo de NF-e funciona (emitir, consultar)
- [ ] Dashboard financeiro carrega corretamente
- [ ] Contratos listam apenas da empresa logada
- [ ] Entregas listam corretamente

### Configuracao do Frontend

O frontend precisa configurar `app.current_empresa_id` no contexto do Supabase para que as policies funcionem. Opcoes:

1. **Via RPC no login:** `supabase.rpc('set_empresa_context', { empresa_id: '...' })`
2. **Via custom claim no JWT:** Adicionar `empresa_id` ao token (requer Supabase Auth — Story 7.2)
3. **Via header:** Configurar `x-empresa-id` e usar funcao SQL que le o header

**Recomendacao:** Usar opcao 1 (RPC) como solucao imediata, migrando para opcao 2 quando Story 7.2 (auth server-side) for implementada.

### Verificacao Pos-Deploy

```sql
-- Verificar RLS ativo
SELECT schemaname, tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY tablename;

-- Verificar policies existentes
SELECT tablename, policyname, permissive, roles, cmd, qual
FROM pg_policies
WHERE schemaname = 'public';
```
