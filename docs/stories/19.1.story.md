## Story
**Como** guardião da integridade de dados,
**Quero** adicionar a coluna `deleted_at` nas tabelas `contas_receber` e `extratos`,
**Para** habilitar o padrão de soft-delete sincronizado (já validado em `conciliacoes` no EPIC-17) nas entidades que ficaram para trás.

## Contexto / Problema
Auditoria read-only no Supabase (mvvsjaudhbglxttxaeop, empresa_id=LARIUCCI) confirmou:
- `conciliacoes` **tem** `deleted_at` (EPIC-17, story 17.6). ✅
- `contas_receber` **NÃO tem** `deleted_at` → erro `42703 column contas_receber.deleted_at does not exist`. ❌
- `extratos` **NÃO tem** `deleted_at` → mesmo erro. ❌

Sem a coluna, essas entidades dependem de hard-delete (DELETE HTTP), frágil a falhas de rede e sem rastro de auditoria — causa-raiz da recorrência.

## Critérios de Aceitação
- **AC1:** *Given* a migration aplicada, *When* consultar `?deleted_at=is.null` em `contas_receber` e `extratos`, *Then* retorna 200 (não mais 42703).
- **AC2:** Migração **aditiva** — nenhum dado existente alterado; zero-downtime.
- **AC3:** Índices parciais `WHERE deleted_at IS NULL` criados para performance de leitura.
- **AC4:** Snapshot do schema imediatamente antes + rollback script disponível.

## Escopo
- **IN:** DDL aditivo (`ALTER TABLE ... ADD COLUMN deleted_at timestamptz NULL`) + índices parciais + snapshot.
- **OUT:** mudança de código (19.2/19.3), reconciliação de dados (19.5), dedupe de extratos (19.4).

## Requisitos Funcionais
- **FR-19.1.1:** `ALTER TABLE contas_receber ADD COLUMN deleted_at timestamptz NULL`.
- **FR-19.1.2:** `ALTER TABLE extratos ADD COLUMN deleted_at timestamptz NULL`.
- **FR-19.1.3:** `CREATE INDEX ... ON <tabela> (empresa_id) WHERE deleted_at IS NULL`.
- **FR-19.1.4:** Snapshot do schema antes; rollback documentado (`DROP COLUMN deleted_at`).
