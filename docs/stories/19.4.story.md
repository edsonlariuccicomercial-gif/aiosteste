# Story 19.4 — Idempotência de "Recuperar Extrato" + Dedupe

**Epic:** EPIC-19 — Correção Definitiva de Sincronização de Exclusões
**Status:** Draft
**Prioridade:** P1
**Risco:** MÉDIO | **Complexidade:** M
**Executor:** @dev + @data-engineer
**Resolve:** ② — extratos recuperados duplicados (1 PC mostra 1, outros mostram 2).

## Story
**Como** usuário da conciliação bancária,
**Quero** que a recuperação de extrato não crie duplicatas e que todos vejam o mesmo número de extratos,
**Para** que a tela de conciliação bancária seja consistente entre todos os PCs.

## Contexto / Problema
A tabela `extratos` tem 2 registros `ext-recovered` com IDs distintos baseados em `Date.now()`:
- `ext-recovered-1781016143488` (total 53)
- `ext-recovered-1781009001859` (total 66)

A rotina "recuperar extrato" (js/gdp-core.js, lógica `ext-recovered` ~L1090-1110 e `registrarExtrato` ~L2540) gera novo `id` a cada execução → **não idempotente** → duplicata. O `sync_data` legado de extratos está **vazio** (não é dupla-fonte ativa; é duplicação real + cache stale).

## Critérios de Aceitação
- **AC1:** *Given* a recuperação de extrato executada 2× para a mesma conta/período, *When* verificada, *Then* existe apenas 1 registro (idempotente).
- **AC2:** *Given* o dedupe aplicado, *When* qualquer PC abre conciliação bancária, *Then* todos veem o MESMO número de extratos recuperados.
- **AC3:** Nenhuma conciliação fica órfã — itens do extrato duplicado re-vinculados ao extrato canônico (`extrato_id` atualizado).
- **AC4 (não-regressão):** importação de extrato real (não-recuperado) continua funcionando.

## Escopo
- **IN:** id idempotente para extrato recuperado; soft-delete de extratos (usa coluna da 19.1); dedupe dos 2 `ext-recovered` atuais (re-vínculo + soft-delete do duplicado).
- **OUT:** contas a receber (19.3), reconciliação de saldo (19.5).

## Requisitos Funcionais
- **FR-19.4.1:** ID de extrato recuperado determinístico (ex.: hash de `conta_financeira` + período/data) — re-execução não cria duplicata.
- **FR-19.4.2:** Soft-delete (`deleted_at`) aplicável a extratos no código (leitura filtra, exclusão marca).
- **FR-19.4.3 (@data-engineer):** Deduplicar os 2 `ext-recovered` atuais: eleger 1 canônico, atualizar `extrato_id` das conciliações vinculadas ao duplicado para o canônico, soft-delete do duplicado. Snapshot antes.

## Arquivos
- `js/gdp-core.js` (`registrarExtrato`, lógica `ext-recovered` ~L1090-1110, L2540+)
- `gdp-api.js` (extratos — `deleted_at` no mapeamento/filtro)
- Script de dedupe controlado (@data-engineer)

## Dependências
**Bloqueada por 19.1** (coluna `deleted_at` em extratos). Dedupe (FR-19.4.3) após 19.2 (filtro de cache ativo) para não reaparecer.
