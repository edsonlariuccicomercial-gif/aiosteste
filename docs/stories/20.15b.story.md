## Problema

Dois navegadores/máquinas logados na mesma empresa (LARIUCCI) mostram **dados divergentes de forma persistente** (ex.: 120 vs 119 notas, 13 vs 11 pedidos entregues), mesmo após Ctrl+Shift+R. A causa não é o cache de arquivos — é o **localStorage de dados que fica "preso"** e resiste a aceitar a versão da nuvem.

A **Parte A** (Story 20.15, já entregue) deu um botão "Forçar Sync" sob demanda — resolve o sintoma quando o usuário percebe a divergência. Esta story (**Parte B**) ataca a **causa-raiz**: fazer a convergência acontecer **automaticamente**, sem intervenção manual.

## Causa Raiz — 5 portões em `js/gdp-core.js` (função `cloudLoadAll` + `saveWrappedArray`)

A reconciliação local↔nuvem passa por uma cadeia de guardas que, combinadas, impedem a nuvem mais recente de sobrescrever um local "tão completo quanto":

| # | Portão | Local aprox. | Comportamento atual |
|---|--------|--------------|---------------------|
| 1 | **Bloqueio por contagem top-level** | L460-463 | `if (localHasContent && !cloudHasMoreContent) continue` — `cloudHasMoreContent = cloudItems >= localItems` (L452). Se local tem MAIS entries, nuvem é ignorada. |
| 2 | **Bloqueio por itens nested + merge** | L466-480 | Se local tem mais `.itens` internos, faz merge preservando local (`mergeArraysPreservingItens`). |
| 3 | **Dirty window 5s** | L482-487 | `if (msSinceLocalSave < 5000) continue` — ignora nuvem se houve save local nos últimos 5s. |
| 4 | **Condição de sobrescrita composta** | L492-494 | `shouldWrite` exige `(isSharedKey && cloudHasMoreContent && cloudHasMoreDeepContent) \|\| (cloudTime > localTime && cloudHasMoreDeepContent) \|\| (!localTime && cloudTime===0)`. As 3 condições simultâneas raramente batem quando local==nuvem em contagem. |
| 5 | **Skip de save no boot** | `saveWrappedArray` ~L993 | `if (_gdpBootInProgress) return` — mudanças feitas durante/logo após o boot não sobem para a nuvem. |

**Efeito combinado:** cada navegador acumula um estado local que "empata" com a nuvem em contagem mas difere em conteúdo, e nenhuma das condições de sobrescrita dispara → divergência crônica.

## Princípio da Correção

A **nuvem (Supabase) é a fonte de verdade** quando é mais recente (`cloudTime > localTime`). A calibração deve permitir que a nuvem mais recente vença **sem exigir as 3 condições de contagem simultâneas**, MAS preservando a proteção legítima contra perda de itens locais que de fato existem só no local (pedidos com `.itens` mais completos ainda não sincronizados).

> **Observação de segurança (validada 2026-06-16):** filas de retry vazias, local == nuvem na inspeção; a nuvem restaurou os dados corretos sem perda. A proteção dos portões 1/2 existe por um motivo real (incidente passado de cloud zerado) — **não removê-la, e sim torná-la condicional ao timestamp**.

## Acceptance Criteria (com cenários Given/When/Then)

### AC1 — Nuvem mais recente vence em empate de contagem
```
DADO que local e nuvem têm a MESMA quantidade de entries (ex.: 120 notas cada)
  E o conteúdo difere (uma nota distinta entre eles)
  E cloudTime > localTime (nuvem foi salva depois)
QUANDO o sync roda (boot ou Forçar Sync)
ENTÃO a nuvem sobrescreve o local
  E os dois navegadores convergem para os mesmos dados
```

### AC2 — Proteção contra perda de itens locais preservada
```
DADO que o local tem um pedido com MAIS .itens que a versão da nuvem
  E cloudTime <= localTime (local é igual ou mais novo)
QUANDO o sync roda
ENTÃO o item local mais completo NÃO é perdido (merge preservando, como hoje)
```

### AC3 — Proteção contra cloud zerado mantida
```
DADO que a nuvem retorna 0 entries para uma chave
  E o local tinha N > 0 entries
QUANDO o sync roda
ENTÃO o local NÃO é zerado (snapshot/restore como em forcarSyncCompleto)
```

### AC4 — Dirty window não trava dado antigo
```
DADO que o usuário NÃO está digitando (último save local > X segundos OU foi um save de boot/sync)
  E a nuvem é mais recente
QUANDO o sync roda
ENTÃO a dirty window de 5s NÃO impede a sobrescrita
  (manter a proteção apenas para edição ativa real do usuário)
```

### AC5 — Mudanças do boot sobem para a nuvem
```
DADO que um dado é alterado durante ou logo após o boot
QUANDO o boot termina
ENTÃO essa alteração é enviada ao Supabase (não fica presa por _gdpBootInProgress)
```

### AC6 — Convergência automática entre 2 navegadores (validação manual)
```
DADO 2 navegadores na mesma empresa (LARIUCCI), partindo de estados divergentes
QUANDO ambos passam por um boot normal (sem clicar Forçar Sync)
ENTÃO mostram contagens IDÊNTICAS (notas, pedidos por status)
```
> Validação manual obrigatória — não automatizável.

### AC7 — Sem operação bloqueante no boot
```
DADO o sistema com carga real (120 notas / 127 pedidos)
QUANDO o boot/sync roda
ENTÃO NÃO há travamento da UI (lição do incidente 2026-06-16)
```

### AC8 — Bump de versão dos JS alterados no HTML

## Riscos

- **CRÍTICO:** mexer nas regras de sync pode causar **perda de dados** se mal feito. As proteções dos portões 1/2/3 existem por causa de incidentes reais (cloud zerado, sobrescrita de edição em andamento). A calibração deve **condicionar ao timestamp**, não remover.
- **CRÍTICO (aprendido 2026-06-16):** nenhuma operação pesada/bloqueante no boot.
- **Obrigatório:** teste de carga real (120 notas / 127 pedidos) + validação em 2 navegadores antes de qualquer deploy.

**Decisões de implementação:**
- **AC3 (não previsto como passo isolado, adicionado por segurança):** ao condicionar os portões 1/4 ao timestamp, identifiquei que uma nuvem **zerada porém com timestamp mais novo** poderia sobrescrever o local e zerá-lo. Adicionei uma **guarda dura absoluta** (`cloudItems === 0 && localItems > 0 → continue`) ANTES dos portões relaxados. Fecha o risco do incidente cloud-zerado.
- **AC4 sem tocar ~106 call-sites:** `saveWrappedArray(key, items, origin)` infere a origem pelo `_gdpBootInProgress` (boot=`system`; pós-boot=`user`, pois o caminho de sync/sanitize usa `localStorage.setItem` direto, não esta função). Helper `gdpMarkUserEdit(key)` exposto para marcação explícita. Default seguro = `system`.
- **AC5/AC7:** push de rede extraído para `_pushNetworkSave()`; saves do boot enfileirados em `_pendingBootSaves` (Set, deduplicado) e reenviados por `_flushPendingBootSaves()` de forma assíncrona ao desligar a flag.
