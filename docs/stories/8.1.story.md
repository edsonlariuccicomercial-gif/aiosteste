# Story 8.1 — Unificar Schema Central de Produtos com Campos do Banco de Precos

**Epic:** Epic 8 — Consolidacao GDP e Fechamento de Gaps Operacionais
**Status:** Ready
**Priority:** P1
**Story Points:** 3

---

## Descricao

A Central de Produtos (`gdp-banco-produtos.js`) e o Banco de Precos (`banco-precos-client.js`) sao dois sistemas sobrepostos que gerenciam dados similares. A Central tem campos de cadastro (descricao, SKU, NCM, unidade, marca) enquanto o Banco tem campos de pricing (custoBase, precoReferencia, custosFornecedor, concorrentes, propostas).

Esta story unifica o schema da Central de Produtos para conter TODOS os campos de ambos os sistemas, criando a base para a migracao de dados (Story 8.2) e deprecacao do Banco de Precos (Story 8.3).

**Referencia SOP:** Etapas 6 (C.A. Referencia) e 7 (Central de Produtos)

---

## Criterios de Aceitacao

- [ ] AC-1: Schema `gdp.produtos.v1` no localStorage contem todos os campos listados em FR-G2-001 da spec
- [ ] AC-2: Campos novos incluem: `custoBase`, `precoReferencia`, `margemAlvo`, `custosFornecedor`, `concorrentes`, `propostas`, `historicoResultados`, `precoReferenciaHistorico`, `taxaConversao`, `grupo`, `fonte`
- [ ] AC-3: Funcao `criarProduto()` em `gdp-banco-produtos.js` aceita os novos campos opcionalmente (retrocompativel)
- [ ] AC-4: Funcao `editarProduto()` em `gdp-banco-produtos.js` permite atualizar os novos campos
- [ ] AC-5: Funcao `getProduto()` retorna o produto com todos os campos (novos e antigos)
- [ ] AC-6: Produtos existentes sem os novos campos continuam funcionando (campos novos sao opcionais com defaults)
- [ ] AC-7: Schema validado — nenhum campo duplicado ou conflitante entre os dois sistemas

---

## Notas Tecnicas

### Arquivos Principais

- **`js/gdp-banco-produtos.js`** — Arquivo principal da Central de Produtos. Contem `criarProduto()`, `editarProduto()`, `renderBancoProdutos()`. O schema atual tem campos: `id`, `descricao`, `sku`, `ncm`, `unidade`, `marca`.

- **`js/banco-precos-client.js`** — SDK do Banco de Precos. Contem classe `BancoPrecos` com metodos como `calcularPreco()`. Schema em `caixaescolar.banco.v1` com campos: `item`, `custoBase`, `precoReferencia`, `custosFornecedor[]`, `concorrentes[]`, `propostas[]`.

### Schema Unificado (FR-G2-001)

```javascript
{
  id: "string",                    // UUID (ja existe)
  descricao: "string",            // Nome normalizado (ja existe)
  sku: "string",                  // SKU interno (ja existe)
  ncm: "string",                  // NCM fiscal (ja existe)
  unidade: "string",              // UN, CX, KG, etc (ja existe)
  marca: "string",                // Marca preferencial (ja existe)
  // --- Campos migrados do Banco de Precos ---
  custoBase: null,                // number — Custo de aquisicao (C.A.)
  precoReferencia: null,          // number — Preco de venda sugerido
  margemAlvo: null,               // number — % margem alvo
  custosFornecedor: [],           // array — Historico de custos
  concorrentes: [],               // array — Precos de concorrentes
  propostas: [],                  // array — Historico de propostas
  historicoResultados: [],        // array — Resultados ganho/perdido
  precoReferenciaHistorico: null, // number — Media de precos ganhos
  taxaConversao: null,            // number — % de propostas ganhas
  grupo: "",                      // string — Grupo de despesa
  fonte: "",                      // string — Origem dos dados
}
```

### Abordagem

1. Adicionar defaults para os novos campos em `criarProduto()`
2. Garantir que `editarProduto()` faz merge sem sobrescrever campos existentes
3. Adicionar funcoes helper: `getProdutoComDefaults(produto)` para garantir que produtos antigos retornam com os campos novos preenchidos com defaults
4. NAO alterar dados existentes em localStorage — apenas garantir que o codigo aceita os novos campos

## Riscos

| Risco | Probabilidade | Mitigacao |
|-------|:-------------:|-----------|
| Conflito de nomes entre campos dos dois sistemas | Baixa | Schema unificado ja mapeado na spec; `item` vira alias de `descricao` |
| localStorage excede cota com campos extras | Baixa | Campos novos sao opcionais e comecam como `null`/`[]` — tamanho minimo |
| Produtos existentes quebram com novo schema | Media | `getProdutoComDefaults()` garante retrocompatibilidade |
