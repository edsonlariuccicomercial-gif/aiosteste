# Story 1.7 — GDP Pedidos: Abas de Status Entrega, Fixes de Edição, Unidades e Totais

**Epic:** GDP — Gestão de Pedidos
**Status:** Ready for Review
**Branch:** `feat/caixa-escolar-fase3-sgd`
**Arquivo principal:** `painel-caixa-escolar/squads/caixa-escolar/dashboard/gdp-contratos.html`

---

## Contexto

A seção Pedidos do GDP apresenta vários problemas: pedidos entregues desaparecem da lista (sem aba de status correspondente), o modo edição não mostra preço unitário, adicionar produtos busca na fonte errada (Inteligência ao invés do contrato), as unidades do contrato não propagam corretamente para pedidos via catálogo, e há erro de cálculo de total em itens com quantidade decimal.

---

## Dependencies

- Stories 1.1–1.6 devem estar merged (mesma branch, mesmo arquivo)
- AC-8 (fix SKU catálogo) deve ser implementado ANTES de AC-9 (refactor SKU) — AC-9 depende de AC-8 para garantir que skuVinculado é priorizado
- Nenhuma dependência externa

## Acceptance Criteria

### AC-1: Novas abas de status "Entregue" e "Não Entregue" na seção Pedidos
- [x] Adicionar aba **"Entregue"** (badge verde) no `renderPedidosStatusTabs()` — filtra pedidos com status `entregue` ou `concluido`
- [x] Adicionar aba **"Não Entregue"** (badge vermelho) — filtra pedidos com status `nao_entregue`, `cancelado`, `devolvido`
- [x] As novas abas devem aparecer APÓS as abas existentes (Em Aberto, Agendado, Preparando Envio, Pronto para Envio, Faturado)
- [x] Contagem de pedidos exibida no badge de cada aba (como já funciona nas abas existentes)

### AC-2: Valor total dos pedidos por aba
- [x] Em cada aba de status, exibir o valor total (soma de `totalGeral` dos pedidos naquela aba) formatado em BRL
- [x] Valor exibido ao lado do badge de contagem ou logo abaixo da aba ativa
- [x] Atualizar valor ao trocar de aba ou ao filtrar por contrato/data

### AC-3: Pedidos entregues NÃO devem desaparecer da lista
- [x] Ao concluir entrega de um pedido, o pedido deve permanecer na seção Pedidos com status `entregue`
- [x] O pedido deve aparecer na aba "Entregue" (AC-1) ao invés de sumir
- [x] Verificar a função que atualiza status pós-entrega e garantir que NÃO remove o pedido do array `gdp.pedidos.v1`
- [x] O pedido entregue deve ser visualizável — ao clicar no protocolo, o card de detalhes abre normalmente

### AC-4: Coluna "Preço Unitário" editável no card de detalhes do pedido
- [x] Na função `verPedidoDetalhe()`, ao clicar em "Editar", a coluna "Preço Unit." deve exibir um `<input type="number">` editável (atualmente é read-only)
- [x] Ao alterar o preço unitário, recalcular o subtotal da linha e o total geral do pedido em tempo real
- [x] Ao salvar, persistir o novo preço unitário no objeto do pedido em localStorage

### AC-5: Fix — Total desconfigurado em itens com quantidade decimal (ex: "Extrato de tomate 1,02 Kg")
- [x] Verificar o pedido da escola "Caixa Escolar da Escola Estadual Lauriston Souza" — item "Extrato de tomate 1,02 Kg - Elefante"
- [x] Garantir que cálculo `qtd * precoUnitario` usa `parseFloat()` e arredonda para 2 casas decimais: `Math.round(subtotal * 100) / 100`
- [x] Aplicar arredondamento consistente em TODAS as funções de cálculo de total (`renderPedidos`, `verPedidoDetalhe`, `salvarPedido`)
- [x] Verificar se `valorTotal` armazenado no pedido está correto — se não, recalcular ao renderizar

### AC-6: Fix — Adicionar produtos ao pedido deve buscar no contrato, não na Inteligência
- [x] Na função `buscarProdutoPedido()`, quando o pedido está vinculado a um contrato, buscar produtos do contrato (`contrato.itens`) ao invés de `estoqueIntelProdutos`
- [x] Se o pedido NÃO tem contrato vinculado, manter busca em `estoqueIntelProdutos` como fallback
- [x] Ao selecionar item do contrato, propagar: descrição, unidade, preço unitário, SKU, NCM do item do contrato
- [x] Exibir indicação visual de que o produto veio do contrato (ex: badge "Contrato" ao lado do nome)

### AC-7: Fix — Unidades do contrato não propagam corretamente para pedidos via Catálogo
- [x] Na função `abrirCatalogoEscolar()`, ao carregar itens do contrato, usar a unidade salva no contrato (não defaultar para `'un'`)
- [x] Se o usuário editou unidades na página de detalhes do contrato, essas edições devem ser persistidas corretamente no `contrato.itens[].unidade`
- [x] Ao criar pedido via catálogo, propagar a unidade correta do item do contrato para o pedido
- [x] Verificar se a função de salvar edição de unidade no contrato está persistindo no localStorage (`gdp.contratos.v1`)
- [x] Garantir que edições de unidade no contrato se refletem em novos pedidos criados via catálogo APÓS a edição

### AC-8: Fix — Pedidos do catálogo usam SKU errado (auto-gerado ao invés do vinculado)
- [x] Em `criarPedidoCatalogo()` (linha ~5209): alterar `sku: item.sku || ''` para `sku: item.skuVinculado || item.sku || ''`
- [x] Garantir que a unidade também priorize a do produto vinculado: `unidade: item.unidadeVinculada || item.unidade || 'un'`
- [x] Em `gdp-portal.html`, verificar se o portal tenta resolver `skuVinculado` quando o `sku` do pedido não bate com nenhum produto Intel
- [x] Caso de teste: contrato `CTR-20260330-0020` (Maestro Josino de Oliveira) — todos os itens vinculados devem gerar pedidos com SKU da Inteligência

### AC-9: Simplificar SKU — não gerar SKU automático no contrato, preencher somente na vinculação
- [x] Remover geração automática de SKU em `buildAutoSku()` / `normalizeInternalSku()` durante import/criação do contrato — campo `sku` do item deve ficar vazio (ou exibir "—")
- [x] O campo `sku` do item do contrato só é preenchido quando o usuário vincula o item a um produto da Inteligência (`selecionarVincularGDPIntel()`)
- [x] Na vinculação, gravar o SKU da Inteligência diretamente em `item.sku` (além de `item.skuVinculado`)
- [x] No portal escolar e nas demandas, continuar usando o SKU vinculado (comportamento atual preservado)
- [x] Na tabela de itens do contrato, exibir badge visual: "Sem SKU" (cinza) quando não vinculado, SKU real quando vinculado

## Risks

- **Regressão em pedidos existentes:** alteração no status flow pode afetar pedidos já criados (mitigação: não alterar pedidos existentes, apenas filtro de exibição)
- **Propagação de unidades:** corrigir propagação pode afetar pedidos futuros mas não retroativos (mitigação: documentar que pedidos antigos mantêm unidade original)
- **Busca de produtos:** mudar fonte de busca para contrato pode causar confusão se contrato não tem todos os itens (mitigação: fallback para Inteligência)
- **Refactor SKU (AC-9):** remover geração automática pode afetar contratos existentes que já têm SKU auto-gerado (mitigação: manter SKUs existentes, só parar de gerar para novos itens; AC-8 fix de priorização garante que skuVinculado é usado)
