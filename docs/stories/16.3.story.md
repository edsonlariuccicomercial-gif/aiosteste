# Story 16.3 — Persistência de Pedidos (Race Condition Realtime)

**Epic:** EPIC-16 — Integridade Financeira Transacional + Persistência de Pedidos
**Status:** Done
**Prioridade:** P1 (independente — pode ir em paralelo)
**Risco:** MÉDIO | **Complexidade:** M
**Resolve sintomas:** B1, B2

## Story
**Como** gestor de pedidos,
**Quero** que alterações de status e data prevista de entrega persistam, e que faturar um pedido o marque automaticamente como "faturado",
**Para** que o sistema não desfaça minhas atualizações sozinho.

## Contexto / Problema
- **B1:** edições de `status` e `dataPrevista` revertem sozinhas. O webhook realtime UPDATE chega na "dirty window" de 5s e aceita o registro stale do Supabase (`remoteTs > localTs` em `handleEntityChange` L173-176), sobrescrevendo a edição local. O save é fire-and-forget (`savePedidos`, gdp-core.js L1345).
- **B2:** `registrarNotaFiscalSaida` (gdp-notas-fiscais.js ~L1057) seta `status = "faturado"` mas a mesma race condition reverte; valor precisa estar no `normalizePedidoStatus`.

## Critérios de Aceitação
- **AC1:** *Given* o gestor altera o status de um pedido, *When* aguarda 30s, *Then* o status permanece (não reverte).
- **AC2:** *Given* o gestor altera a data prevista de entrega, *When* aguarda 30s e recarrega, *Then* a data persiste.
- **AC3:** *Given* um pedido em aberto, *When* o gestor fatura (emite NF), *Then* o status muda automaticamente para `faturado` e persiste.
- **AC4:** *Given* `faturado` setado, *When* o webhook realtime chega com registro mais antigo, *Then* a edição local NÃO é sobrescrita.

## Escopo
- **IN:** `updated_at` síncrono antes do webhook; dirty-window protegendo `status`/`dataPrevista`; normalizador inclui `faturado`; persistência do faturado com await/retry.
- **OUT:** financeiro/caixa (16.1, 16.2, 16.4).

## Dev Notes (Dex)
**Causa-raiz exata confirmada:** `savePedidos()` setava `_lastLocalSave` (inicia dirty window) mas NÃO atualizava o `updated_at` do pedido. No webhook, `handleEntityChange` comparava `remoteTs > localTs` com um `localTs` defasado → o registro stale do Supabase sempre vencia e revertia a edição.

**Correção (2 camadas, determinística):**
1. **`savePedidos` (gdp-core.js):** carimba `updated_at`/`updatedAt` SÍNCRONO em cada pedido ANTES do save. O timestamp local passa a refletir a edição.
2. **`handleEntityChange` (gdp-realtime.js):** durante a dirty window, NÃO sobrescreve item que já existe localmente (o webhook é eco do próprio save, possivelmente stale). Itens novos de outra máquina ainda entram. Elimina a race de forma determinística — não depende de comparação de timestamp frágil.

- **AC1/AC2 (B1):** status e dataPrevista persistem — webhook não reverte mais.
- **AC3 (B2):** `registrarNotaFiscalSaida` já setava `status="faturado"` + `savePedidos()`; agora persiste (mesma correção). `normalizePedidoStatus` JÁ tratava `faturado` corretamente (nenhuma mudança necessária).
- **AC4:** registro remoto mais antigo na dirty window é ignorado para itens existentes.

**Não-regressão:** sync de outras máquinas preservado (INSERT de itens novos + UPDATE fora da dirty window inalterados). Sintaxe validada.
