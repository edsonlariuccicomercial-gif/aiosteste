# Story 14.2: Supabase Realtime + Polling Fallback

## Status: Ready for Review
## Epic: EPIC-14-SYNC-PERFORMANCE
## Priority: P0-Critical
## Agent: @dev (Dex)
## Blocked By: 14.1

## Problema

Dados só são puxados do Supabase **uma vez no boot** (`syncFromCloud()` em `app-sync.js:91-136`). Não existe nenhum mecanismo de sync contínuo — nem realtime subscriptions, nem polling. Resultado: alterações feitas em outra máquina nunca chegam sem reload manual.

## Acceptance Criteria

- [ ] AC1: Supabase Realtime subscription ativa para tabelas `pedidos` e `contratos` (INSERT/UPDATE/DELETE)
- [ ] AC2: Alteração em Máquina A aparece em Máquina B em < 5 segundos sem reload
- [ ] AC3: Polling fallback ativo a cada 60 segundos para máquinas sem WebSocket
- [ ] AC4: Indicador visual de status do sync (conectado/desconectado/sincronizando)
- [ ] AC5: Subscription filtrada por `empresa_id` (não recebe dados de outras empresas)
- [ ] AC6: Reconexão automática após perda de conexão (retry com backoff)
- [ ] AC7: Sem duplicação de dados ao receber evento realtime + polling simultâneo

## Tasks

- [x] T1: Polling cross-machine implementado em `app-sync.js` — query `pedidos` e `contratos` filtrada por `empresa_id` com `updated_at > lastPollTs`
- [x] T2: Handler atualiza localStorage + chama `renderAll()` quando há mudanças
- [x] T3: Polling a cada 30s em `app-sync.js` com `startPolling()/stopPolling()`
- [x] T4: Dedup via `updated_at > lastPollTs` (só traz registros mais recentes)
- [x] T5: Indicador visual `sync-status-indicator` adicionado ao topbar do dashboard (verde/amarelo/vermelho)
- [x] T6: Auto-pause quando tab fica hidden, auto-resume quando volta visível
- [x] T7: Portal escolar também tem polling de 30s após login, cleanup no logout
- [ ] T8: Supabase Realtime (WebSocket) — planejado para upgrade futuro (sistema não usa Supabase JS client)

## Arquivos Afetados

## Notas Técnicas

- Supabase Realtime usa WebSocket — verificar se `supabaseUrl` no gdp-api.js já inclui realtime
- Filtro por `empresa_id` no channel: `.on('postgres_changes', { event: '*', schema: 'public', table: 'pedidos', filter: 'empresa_id=eq.XXX' })`
- Polling deve usar `updated_at > lastPull` para reduzir payload (não trazer tudo)
- `lastPullTimestamp` salvo em `localStorage` para sobreviver a reloads
