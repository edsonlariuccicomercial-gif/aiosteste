## Problema

Em **Financeiro > Contas a Receber > Contas Vencidas**, ao tentar enviar uma **cobrança via WhatsApp** para a escola **"América"**, o sistema exibe a mensagem **"Telefone do cliente não encontrado. Cadastre no módulo Clientes."** — porém **esse cliente TEM telefone cadastrado**.

## Causa Raiz (evidência de código — @analyst)

É **consequência direta do PROB-1** (normalização de acento). A função `_buscarTelefoneCliente()` em `gdp-core.js:2013-2020` casa o cliente por **nome exato com `.toLowerCase()`, SEM remover acento**:

```javascript
function _buscarTelefoneCliente(nomeCliente) {
  if (!nomeCliente) return null;
  const nome = nomeCliente.trim().toLowerCase();          // <- não remove acento
  const cliente = (usuarios).find(u =>
    (u.nome || '').trim().toLowerCase() === nome          // <- match exato, accent-sensitive
  );
  return cliente ? (cliente.telefone || '') : '';
}
```

Quando `conta.cliente = "America"` (sem acento, como gravado na conta a receber) e o cadastro é `usuario.nome = "América"` (com acento) ou `"Escola América"`, o `.find()` **falha** porque `"america" !== "américa"`. O lookup retorna vazio → `_formatarTelefoneWpp()` retorna `null` → dispara o falso "telefone não encontrado".

Chamador: `enviarLembreteConta(contaId, canal)` em `gdp-core.js:2052-2068`.

**Agravante de design (fora de escopo desta correção):** a junção cliente↔conta é feita por **string de nome**, não por `client_id` — frágil a variações de acento/casing/espaço. Esta story corrige o sintoma (normalização); a junção por ID fica como débito técnico.

## Princípio da Correção

Aplicar `window.normalizeSearch` (corrigido na 20.16) **nos dois lados** da comparação de nome em `_buscarTelefoneCliente()`, tornando o match accent/casing/pontuação-insensitive — exatamente como as demais buscas do sistema.

## Acceptance Criteria (Given/When/Then)

### AC1 — Cobrança encontra telefone independente de acento
```
DADO uma conta a receber com cliente "America" (sem acento)
  E um cadastro de cliente "América" (com acento) COM telefone
QUANDO o usuário envia cobrança via WhatsApp
ENTÃO o telefone é encontrado e a cobrança abre o WhatsApp (não exibe "telefone não encontrado")
```

### AC2 — Match insensitive a casing e espaços
```
DADO variações como "escola américa", "ESCOLA AMERICA", " América "
QUANDO o lookup de telefone roda
ENTÃO todas casam com o cadastro correspondente
```

### AC3 — Cliente realmente sem telefone ainda avisa
```
DADO um cliente que de fato NÃO tem telefone cadastrado
QUANDO o usuário tenta a cobrança
ENTÃO a mensagem "Telefone do cliente não encontrado" ainda aparece (comportamento legítimo preservado)
```

### AC4 — Sem regressão no formato do número WhatsApp
```
DADO um telefone encontrado
QUANDO formatado por _formatarTelefoneWpp
ENTÃO o número WhatsApp continua sendo gerado corretamente (DDI/DDD)
```

### AC5 — Bump de versão dos JS alterados no HTML
