# Story 6.1 — Persistir Resultados SGD em Supabase

**Epic:** Epic 6 — Inteligência de Preços: Fechamento de Loops
**Status:** Done
**Priority:** P0
**Story Points:** 3

---

## Problema

Os resultados de orçamentos SGD (ganho/perdido) são armazenados **apenas em localStorage** (`RESULTADOS_STORAGE_KEY`). Se o usuário limpar o browser, trocar de máquina, ou o storage corromper, todo o histórico de resultados é perdido. Isso impede análise de longo prazo, cálculo confiável de taxa de conversão, e aprendizado do sistema de preços.

## Scope

### IN
- Criar migration Supabase `004_resultados_orcamento.sql` com tabela `resultados_orcamento`
- Alterar `salvarResultado()` em `app-results.js` para persistir em Supabase além de localStorage
- Alterar `checarStatusSgd()` para persistir resultado automático em Supabase
- Carregar resultados do Supabase no boot (com fallback localStorage)
- Migrar resultados existentes do localStorage para Supabase (one-time)

### OUT
- Dashboard de analytics de resultados (Story 6.4)
- Integração com price-history consolidado (Story 6.3)

## Acceptance Criteria

### AC-1: Migration Supabase
- [x] Criar `004_resultados_orcamento.sql` com tabela:
  - `id`, `empresa_id`, `orcamento_id`, `resultado` (ganho/perdido/enviado)
  - `valor_proposta`, `valor_vencedor`, `fornecedor_vencedor`, `motivo_perda`
  - `escola`, `municipio`, `sre`, `objeto`, `sku`
  - `itens` (JSONB — array de itens do orçamento com preços)
  - `created_at`, `updated_at`
  - RLS: empresa_id = auth.uid()
- [x] Índices: `empresa_id`, `resultado`, `orcamento_id`
- [x] Trigger `trg_updated_at` reutilizando função existente

### AC-2: Dual-write em salvarResultado()
- [x] Após salvar em localStorage, fazer upsert em Supabase `resultados_orcamento`
- [x] Se Supabase falhar, continuar com localStorage (graceful degradation)
- [x] Log no console: "Resultado salvo em Supabase" ou "Fallback localStorage"
- [x] `checarStatusSgd()` também persiste resultado automático em Supabase

### AC-3: Carregar resultados do Supabase no boot
- [x] No boot do app, buscar resultados de Supabase por empresa_id
- [x] Merge com localStorage: se mesmo `orcamento_id` existe em ambos, usar registro com `updated_at` mais recente
- [x] Se resultado em Supabase difere de localStorage para mesmo `orcamento_id`, Supabase prevalece e localStorage é atualizado
- [x] Se Supabase indisponível, usar localStorage normalmente

### AC-4: Migração one-time de dados existentes
- [x] Detectar se há resultados em localStorage que não existem em Supabase
- [x] Migrar automaticamente na primeira execução
- [x] Marcar migração como feita em localStorage (`resultados.migrated.v1 = true`)
