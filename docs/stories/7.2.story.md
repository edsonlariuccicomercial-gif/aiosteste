# Story 7.2 — Corrigir Autenticacao: Migrar para Server-Side

**Epic:** Epic 7 — Modernizacao e Remediacao de Divida Tecnica
**Status:** Ready
**Priority:** P0 (CRITICAL — Sprint 0)
**Story Points:** 8

---

## Descricao

A autenticacao atual e **puramente client-side**: um hash SHA-256 hardcoded em `auth.js` que compara a senha digitada. Qualquer pessoa com DevTools pode ver o hash, bypassar a verificacao, ou simplesmente remover o script de autenticacao. Nao existe nenhuma barreira real de acesso ao sistema.

Alem disso, a chave `anon` do Supabase esta exposta em 4+ arquivos JS do frontend, permitindo CRUD direto via API REST do Supabase para qualquer pessoa que inspecione o codigo-fonte.

### Problema de Negocio

- Zero barreira de entrada: qualquer pessoa com a URL acessa o sistema completo
- Hash SHA-256 client-side nao oferece seguranca alguma
- Sem sessoes, sem tokens, sem expiracoes
- Impossivel auditar quem acessou o que

### Solucao

Implementar autenticacao real via **Supabase Auth** (server-side), com:
- Login por email/senha gerenciado pelo Supabase
- Sessao JWT com expiracoes
- `auth.uid()` disponivel para policies RLS
- Frontend nao acessa banco sem sessao valida

## Criterios de Aceitacao

### AC-1: Supabase Auth configurado
- [ ] Supabase Auth habilitado no projeto (email/password provider)
- [ ] Pelo menos 1 usuario criado (admin da empresa LARIUCCI)
- [ ] Configuracoes de seguranca: sessao expira em 1h, refresh token em 7d
- [ ] Email de confirmacao desabilitado inicialmente (simplificar deploy)

### AC-2: Tela de login funcional
- [ ] Nova tela de login (ou adaptacao da existente) usando `supabase.auth.signInWithPassword()`
- [ ] Feedback visual de erro para credenciais invalidas
- [ ] Redirecionamento para dashboard apos login bem-sucedido
- [ ] Botao de logout que chama `supabase.auth.signOut()`

### AC-3: Protecao de rotas no frontend
- [ ] Verificacao de sessao ativa no boot de todas as paginas (`supabase.auth.getSession()`)
- [ ] Se sessao inexistente ou expirada, redirecionar para tela de login
- [ ] Listener de `onAuthStateChange` para detectar logout/expiracoes em tempo real

### AC-4: Backend valida sessao
- [ ] Serverless functions validam o JWT do usuario antes de processar requisicoes
- [ ] Requisicoes sem token valido retornam 401 Unauthorized
- [ ] `service_role` key usada apenas no backend, nunca no frontend

### AC-5: RLS integrado com auth.uid()
- [ ] Policies RLS atualizadas para usar `auth.uid()` em vez de `current_setting('app.current_empresa_id')`
- [ ] Tabela de mapeamento `user_empresa` criada (usuario -> empresa_id) ou custom claim no JWT
- [ ] Funcao helper `get_user_empresa_id()` que retorna empresa_id do usuario logado

### AC-6: Remocao do auth.js legacy
- [ ] Arquivo `auth.js` com hash SHA-256 removido ou desativado
- [ ] Nenhum fluxo de autenticacao client-side permanece ativo
- [ ] Hash hardcoded removido do codigo

### AC-7: Nenhuma regressao funcional
- [ ] Todos os fluxos existentes (pedidos, NF-e, contratos, dashboard) funcionam apos login
- [ ] Dados carregam corretamente para a empresa do usuario logado
- [ ] Operacoes CRUD continuam funcionando normalmente

## Notas Tecnicas

### Arquitetura de Autenticacao (Supabase Auth)

```
[Browser] --login--> [Supabase Auth] --JWT--> [Browser armazena token]
[Browser] --request+JWT--> [Supabase DB/API] --valida JWT--> [Executa com RLS]
[Browser] --request+JWT--> [Serverless Functions] --valida--> [Processa]
```

### Mapeamento Usuario-Empresa

Opcoes para vincular usuario a empresa:

1. **Custom claim no JWT** (recomendado):
   ```sql
   -- Hook que adiciona empresa_id ao JWT
   CREATE OR REPLACE FUNCTION custom_access_token_hook(event jsonb)
   RETURNS jsonb AS $$
   DECLARE
     claims jsonb;
     empresa_id text;
   BEGIN
     SELECT e.id INTO empresa_id
     FROM user_empresa ue
     JOIN empresas e ON e.id = ue.empresa_id
     WHERE ue.user_id = (event->>'user_id')::uuid;
     
     claims := event->'claims';
     claims := jsonb_set(claims, '{empresa_id}', to_jsonb(empresa_id));
     event := jsonb_set(event, '{claims}', claims);
     RETURN event;
   END;
   $$ LANGUAGE plpgsql;
   ```

2. **Tabela de mapeamento simples:**
   ```sql
   CREATE TABLE user_empresa (
     user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
     empresa_id TEXT REFERENCES empresas(id),
     role TEXT DEFAULT 'operador',
     PRIMARY KEY (user_id, empresa_id)
   );
   ```

### Atualizacao das Policies RLS

Apos Story 7.1 (RLS basico), as policies devem ser atualizadas para usar `auth.uid()`:

```sql
-- Exemplo: policy atualizada para usar JWT
CREATE OR REPLACE POLICY clientes_isolation ON clientes
  FOR ALL
  USING (empresa_id = (current_setting('request.jwt.claims', true)::jsonb->>'empresa_id'))
  WITH CHECK (empresa_id = (current_setting('request.jwt.claims', true)::jsonb->>'empresa_id'));
```

### Migracao de Usuarios

Como o sistema e single-tenant atualmente (empresa LARIUCCI), a migracao e simples:
1. Criar 1 usuario no Supabase Auth
2. Criar registro em `user_empresa` vinculando ao empresa_id existente
3. Compartilhar credenciais com o operador atual
