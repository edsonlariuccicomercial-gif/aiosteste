# Story 4.72 — Otimizar sync GDP: batch upsert + lazy load (eliminar 55 requests no boot)

- **Epic:** EPIC-4 — GDP Caixa Escolar
- **Status:** Ready
- **Complexity:** M (Medium)
- **Agent:** @dev
- **Depends on:** None
- **Priority:** High (UX degradado — 29s de bloqueio no boot)

## Descrição

O módulo GDP faz **55 chamadas individuais** ao endpoint `sync_data` do Supabase durante o boot da página. Cada chamada leva ~2.7s (latência rede), resultando em **29 segundos de wall-clock** antes do dashboard ficar interativo. O browser limita a 6 conexões simultâneas por domínio, criando filas sequenciais.

**Medições (@analyst via Playwright):**
- 55 chamadas a `sync_data` (27 uploads + candidatos download + polling)
- 147s acumulados, 29s wall-clock
- Média: 2.679ms por chamada
- Page load: 2.3s (scripts rápidos, APIs lentas)

**Bugs secundários encontrados:**
- `ReferenceError: gdpLog is not defined` em `supabase-auth.js:165` e `gdp-api.js:379`
- 404 em `rpc/set_empresa_context` (RPC inexistente)

## Acceptance Criteria

### AC1: Batch upsert no syncToCloud
- **Given** o módulo GDP precisa sincronizar 27 keys para o cloud
- **When** `syncToCloud()` é chamado
- **Then** faz **1 único POST** com array de items (Supabase suporta batch upsert com `Prefer: resolution=merge-duplicates`)
- **And** o tempo total de upload cai de ~15s para ~3s (1 request)

### AC2: Lazy sync no boot
- **Given** o usuário abre a página GDP
- **When** o boot inicia
- **Then** carrega dados do localStorage imediatamente (render instantâneo)
- **And** `syncFromCloud()` roda em background (setTimeout 0 ou requestIdleCallback) APÓS `renderAll()`
- **And** ao completar o sync, faz re-render apenas se houver dados novos

### AC3: Fix gdpLog undefined
- **Given** `supabase-auth.js` e `gdp-api.js` usam `gdpLog`
- **When** esses scripts carregam
- **Then** `gdpLog` já está definido (garantir `gdp-core.js` carrega antes)
- **Or** usar fallback: `(typeof gdpLog === 'function' ? gdpLog : console.log)`

### AC4: Remover chamada set_empresa_context
- **Given** `rpc/set_empresa_context` retorna 404
- **When** o boot executa
- **Then** essa chamada não é mais feita (remover ou wrap em try-catch silencioso)

## Scope

### IN
- Refatorar `syncToCloud()` em `app-sync.js` para batch upsert
- Mover `syncFromCloud()` para background no boot GDP
- Corrigir ordem de scripts ou adicionar fallback para `gdpLog`
- Remover/silenciar chamada `set_empresa_context`

### OUT
- Mudança no schema Supabase `sync_data`
- Refatoração do polling (Story 14.2) — manter como está
- Migração de dados para tabelas dedicadas

## Análise Técnica (@analyst)

### Arquivo 1: `app-sync.js`
- **Linha 220-250:** `syncToCloud()` — substituir loop de `SYNC_KEYS.map(cloudSave)` por batch POST
- **Linha 46-61:** `cloudSave()` — criar `cloudSaveBatch(items)` que faz 1 POST com array

### Arquivo 2: `gdp-init.js` ou `gdp-contratos.html`
- Mover chamada `syncFromCloud()` para APÓS `renderAll()` com `requestIdleCallback`

### Arquivo 3: `supabase-auth.js` (linha 165)
- Adicionar guard: `if (typeof gdpLog === 'function') gdpLog(...)` ou reordenar script tags

### Arquivo 4: `gdp-api.js` (linha 379)
- Mesmo fix de guard para `gdpLog`

### Referência Supabase batch upsert:
```javascript
// Antes: 27 requests individuais
SYNC_KEYS.map(key => cloudSave(key, data));

// Depois: 1 request batch
const rows = SYNC_KEYS.map(key => ({ user_id, key, data: getData(key), updated_at: now })).filter(Boolean);
await fetch(`${SUPABASE_URL}/rest/v1/sync_data?on_conflict=user_id,key`, {
  method: "POST",
  headers: { ...headers, "Prefer": "return=minimal,resolution=merge-duplicates" },
  body: JSON.stringify(rows) // Array!
});
```

## Risks

- **Medium:** Batch POST pode exceder tamanho máximo de body se keys tiverem payloads grandes
- **Mitigation:** `SYNC_MAX_BYTES` (500KB) já existe — filtrar keys grandes do batch, manter individuais apenas para keys > 100KB
- **Low:** Re-render após sync pode causar flicker
- **Mitigation:** Comparar hash antes de re-render (já existe `_quickHash`)
