# Story 6.5 — Granularizar SKUs: De Categoria para Item Específico

**Epic:** Epic 6 — Inteligência de Preços: Fechamento de Loops
**Status:** Done
**Priority:** P2
**Story Points:** 5

---

## Problema

O sistema atual usa 14 SKUs genéricos por categoria (ex: `ALIM-ARROZ-5KG`, `LIMP-AGUA-SANITARIA-1L`). Isso significa que "Arroz Tipo 1 5kg" e "Arroz Parboilizado 5kg" compartilham o mesmo `custoBase` e `precoSugerido`.

Na prática, itens dentro da mesma categoria têm custos e margens diferentes. Um feijão carioca 1kg custa diferente de feijão preto 1kg, mas ambos caem em `ALIM-FEIJAO-1KG`.

O `object-sku-rules.json` tem 15 regras (1 duplicada) que fazem matching por keywords para 14 SKUs. Para avançar na precisão de precificação, precisamos:
1. Permitir SKUs por item específico (sub-SKU dentro da categoria)
2. Manter backward compatibility com SKUs de categoria
3. Expandir `sku-costs.json` progressivamente

## Scope

### IN
- Implementar sistema de sub-SKU hierárquico: `ALIM-ARROZ-5KG` → `ALIM-ARROZ-TIPO1-5KG`, `ALIM-ARROZ-PARBO-5KG`
- Ajustar `skuForQuote()` para tentar match específico antes de fallback categoria
- Expandir `object-sku-rules.json` com regras mais específicas (top 20 itens por volume)
- Ajustar `sku-costs.json` com custos por item quando disponível
- Fix regra duplicada #15 (ESCR-CADERNO)

### OUT
- Reescrever radar-matcher.js (já funciona bem)
- Importação automática de novos SKUs (manual por agora)
- Machine learning para classificação automática

## Acceptance Criteria

### AC-1: Sistema de sub-SKU hierárquico
- [ ] SKUs seguem padrão: `CATEGORIA-SUBCATEGORIA-DETALHE`
  - Exemplo: `ALIM-ARROZ-TIPO1-5KG`, `ALIM-FEIJAO-CARIOCA-1KG`
- [ ] `findSkuByPrefix()` tenta match mais específico primeiro (mais tokens), depois fallback
- [ ] Backward compatibility: `ALIM-ARROZ-5KG` continua funcionando como fallback

### AC-2: Expandir object-sku-rules.json
- [ ] Remover regra duplicada #15 (Material consumo geral → ESCR-CADERNO)
- [ ] Adicionar regras específicas para top 10 itens mais frequentes do histórico:
  - Arroz tipo 1 5kg, Arroz parboilizado 5kg
  - Feijão carioca 1kg, Feijão preto 1kg
  - Água sanitária 1L, Detergente 500ml
  - Papel higiênico, Papel toalha
  - Caderno brochura, Lápis grafite
- [ ] Cada regra específica deve ter keywords mais restritivas que a regra de categoria

### AC-3: Expandir sku-costs.json
- [ ] Adicionar `custoBase` específico para cada novo sub-SKU
- [ ] Manter items de categoria como fallback (sem remover)
- [ ] Exemplo: `ALIM-ARROZ-TIPO1-5KG: custoBase 24.80` vs `ALIM-ARROZ-PARBO-5KG: custoBase 26.50`

### AC-4: Ajustar skuForQuote() para hierarquia
- [ ] Ordem de matching:
  1. Tentar regras específicas primeiro (mais keywords = mais específico)
  2. Se nenhuma específica, fallback para regra de categoria
  3. Se nenhuma categoria, fallback para hardcoded (como hoje)
- [ ] Log no console: "SKU match: [objeto] → [sku] (específico|categoria|fallback)"

### AC-5: Validar cobertura pós-expansão
- [ ] Rodar `audit-sku-coverage.js` após mudanças
- [ ] Coverage deve manter 100% (nenhuma regressão)
- [ ] Novas regras específicas devem aparecer no log com "(específico)" tag
