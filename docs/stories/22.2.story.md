# Story 22.2 — Persistência ONLINE da visibilidade de módulos (Supabase, por empresa)

**Epic:** EPIC-22 — Módulos por Cliente
**Resolve:** P4 (handoff @analyst 2026-06-22) · **Prioridade:** P1 · **Risco:** ALTO (schema + RLS + sync) · **Complexidade:** M
**Status:** Ready _(condicional: passar pelo Spec Pipeline antes do @dev — schema/RLS)_
**Depende de:** Story 22.1 (Done — fonte única `modulos-acesso.js`)

> ⚠️ **Mudança de design**, não bug de re-render. A Story 22.1 resolveu a *re-aplicação visual* da config na mesma máquina, mas a config **persiste apenas em `localStorage`**. Esta story leva a persistência ao **Supabase**, por empresa, para valer em qualquer navegador/computador/usuário. Recomenda-se passar pelo **Spec Pipeline** (@pm/@architect/@data-engineer) antes do desenvolvimento — envolve schema novo e RLS.

## Contexto

O stakeholder (Edson) relatou (2026-06-22): marcou "deixar só o GDP disponível" e a mudança **só refletiu no próprio navegador** — outras máquinas/usuários continuaram com a config antiga. O **@analyst (Atlas)** confirmou por código: a config nunca vai ao Supabase.

## Causa-raiz (evidência de código)

```
modulos-acesso.js:10      MODULOS_KEY = "nexedu.modulos.acesso"     (localStorage)
modulos-acesso.js:19-32   getAcessoModulos()  → lê só localStorage.getItem
modulos-acesso.js:35-44   setAcessoModulos()  → grava só localStorage.setItem
```

Não há nenhuma escrita/leitura via `gdpApi`/Supabase para esta config. Toda mudança fica presa ao navegador local.

## Decisão do Stakeholder (2026-06-22)

- **Granularidade: por EMPRESA** (`empresa_id`). A escolha de módulos vale para a empresa inteira — todos os usuários/máquinas daquela empresa veem a mesma config. (Confirmado via AskUserQuestion.)
- Mudanças devem ser **online**, refletindo em qualquer navegador/computador/usuário da mesma empresa.

## Requisitos Funcionais

- **FR-22.2.1:** Persistir a config de acesso a módulos no **Supabase**, chaveada por `empresa_id`. (Decisão de schema — tabela nova `empresa_modulos` **ou** coluna JSON em config de empresa existente — fica a cargo de @architect/@data-engineer.)
- **FR-22.2.2:** `getAcessoModulos()` passa a **ler do Supabase** (via `gdpApi`) no init, com **fallback gracioso** ao último cache local (`localStorage`) se o Supabase estiver indisponível. Manter o **default seguro**: sem config salva → todos os módulos visíveis (nunca travar cliente novo).
- **FR-22.2.3:** `setAcessoModulos()` passa a **gravar no Supabase** (via `gdpApi.<...>.save()`), registrando supressão de eco (`_markSelfEcho`) no mesmo padrão dos demais saves, e atualizando o cache local.
- **FR-22.2.4:** Propagar mudanças via **realtime** para sessões abertas de outros usuários/máquinas da mesma empresa → re-executar `aplicarAcessoSidebar()` ao receber update.
- **FR-22.2.5:** Política **RLS** por `empresa_id` (delegar implementação a @data-engineer): cada empresa lê/escreve apenas a própria config.
- **FR-22.2.6:** Migração: a config existente em `localStorage` do usuário, se houver, é **semeada** no Supabase na primeira escrita (ou ignorada em favor do que estiver no banco, conforme decisão do @architect).

## Critérios de Aceitação

- **AC1:** *Given* eu altero a visibilidade de módulos num navegador, *When* outro usuário/máquina da mesma empresa recarrega (ou recebe realtime), *Then* a mesma config é aplicada.
- **AC2:** *Given* empresas distintas, *When* cada uma configura seus módulos, *Then* uma não afeta a outra (isolamento por `empresa_id` / RLS).
- **AC3:** *Given* nenhuma config salva no banco para a empresa, *When* qualquer usuário entra, *Then* todos os módulos ficam visíveis (default seguro).
- **AC4:** *Given* o Supabase indisponível, *When* a página carrega, *Then* usa o último cache local sem quebrar (fallback gracioso), e volta a sincronizar quando online.
- **AC5:** *Given* uma mudança de config, *When* salva, *Then* não é revertida por eco realtime na própria sessão (supressão de eco).

## Escopo

- **IN:** persistência da config no Supabase por empresa, leitura/escrita via gdpApi, fallback local, realtime, RLS.
- **OUT:** separação dos módulos em produtos vendáveis e API de integração do núcleo (Fase 2/3 do EPIC-22 — @architect, não bloqueante); UI de marcação (já existe na Story 22.1).
