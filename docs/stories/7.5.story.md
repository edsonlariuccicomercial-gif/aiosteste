# Story 7.5 — Remover Chaves Expostas do Frontend

**Epic:** Epic 7 — Modernizacao e Remediacao de Divida Tecnica
**Status:** Ready
**Priority:** P0 (CRITICAL — Sprint 0)
**Story Points:** 3

---

## Descricao

A chave `anon` do Supabase esta exposta em **4 ou mais arquivos JavaScript** do frontend. Sem autenticacao real (TD-001) e sem RLS (TD-002), essa chave permite CRUD irrestrito em todas as tabelas. Alem disso, credenciais do SGD (CNPJ + senha de fornecedor no portal governamental) estao armazenadas em `localStorage` sem criptografia.

**Apos as Stories 7.1 (RLS) e 7.2 (Auth server-side) estarem implementadas**, a chave `anon` fica segura (pois RLS bloqueia acesso sem sessao autenticada). Porem, e necessario:

1. Remover exposicoes desnecessarias da `service_role` key (se existirem no frontend)
2. Centralizar a configuracao do Supabase em um unico ponto
3. Migrar credenciais SGD de localStorage para server-side

### Problema de Negocio

- `service_role` key no frontend = bypass total de RLS (acesso administrativo irrestrito)
- Credenciais SGD em localStorage podem ser roubadas por extensoes maliciosas, XSS ou acesso fisico
- Chave `anon` duplicada em 4+ arquivos dificulta rotacao de chaves

### Solucao

1. Auditar TODOS os arquivos JS/HTML para encontrar chaves Supabase
2. Remover qualquer `service_role` key do frontend (CRITICO)
3. Centralizar `anon` key em um unico arquivo de configuracao
4. Migrar credenciais SGD para env vars do servidor (serverless functions)

- [ ] Serverless functions continuam usando service_role via env vars

- [ ] `localStorage` nao contem mais credenciais de terceiros (SGD)

### Migracao de Credenciais SGD

**Antes (inseguro):**
```javascript
// Frontend salva em localStorage
localStorage.setItem('sgd_cnpj', cnpj);
localStorage.setItem('sgd_password', senha);

// Frontend envia diretamente
const response = await fetch('/api/sgd-proxy', {
  body: JSON.stringify({ cnpj: localStorage.getItem('sgd_cnpj'), ... })
});
```

**Depois (seguro):**
```javascript
// Frontend: formulario de configuracao salva em Supabase (protegido por RLS)
await supabase.from('empresa_config')
  .upsert({ empresa_id, sgd_cnpj: cnpj, sgd_password: encryptedPassword });

// Serverless function: busca credenciais do banco (service_role)
// api/sgd-proxy.js
const { data } = await supabase.from('empresa_config')
  .select('sgd_cnpj, sgd_password')
  .eq('empresa_id', empresaId)
  .single();
```

### Centralizacao da Config Supabase

```javascript
// supabase-config.js (UNICO ponto de configuracao)
const SUPABASE_URL = 'https://xxx.supabase.co';
const SUPABASE_ANON_KEY = 'eyJ...'; // anon key (segura com RLS ativo)

export const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY, {
  auth: {
    persistSession: true,
    autoRefreshToken: true
  }
});
```

### Checklist de Rotacao de Chaves (para futuro)

Apos centralizacao, a rotacao de chaves se torna trivial:
1. Gerar nova key no Supabase Dashboard
2. Atualizar `supabase-config.js` (frontend)
3. Atualizar env vars no Vercel (backend)
4. Deploy
