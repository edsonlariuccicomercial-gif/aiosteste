# Story 6.3 — Integrar Price History com Resultados Reais (Ganho/Perdido)

**Epic:** Epic 6 — Inteligência de Preços: Fechamento de Loops
**Status:** Done
**Priority:** P1
**Story Points:** 3

---

## Problema

O pipeline `build-price-history.js` → `summarize-price-history.js` constrói o histórico de preços lendo **apenas `quotes.json`** (propostas coletadas). Ele ignora completamente os resultados reais (ganho/perdido) que estão em localStorage via `alimentarBancoComResultado()`.

Isso significa que:
- O histórico trata todas as propostas igualmente (não diferencia se ganhou ou perdeu)
- As estatísticas (min/median/max) misturam preços vencedores com preços perdedores
- O `suggestedUnitPriceForSku()` usa referências sem saber se aquele preço realmente funcionou
- A `taxaConversao` calculada em `app-results.js` não alimenta o pipeline de sugestão

## Scope

### IN
- Enriquecer `build-price-history.js` para marcar entries com tipo (proposta/ganho/perdido)
- Ler resultados de Supabase `resultados_orcamento` (criada na Story 6.1) e correlacionar com quotes
- Enriquecer `summarize-price-history.js` com estatísticas separadas por tipo
- Separar: `medianGanho` vs `medianPerdido` vs `medianGeral` por SKU

### OUT
- Engine de preço sugerido v2 (Story 6.4 — consome estas métricas)
- Dashboard visual de resultados (futuro)

### DEPENDS ON
- **Story 6.1** — Resultados em Supabase (source de dados para enriquecimento)
- **Story 6.2** — Tabela `preco_historico` (registros tipo nf_entrada disponíveis)

---

## Acceptance Criteria

### AC-1: build-price-history.js enriquecido com resultados
- [ ] Ler `resultados_orcamento` do Supabase (fallback: localStorage `RESULTADOS_STORAGE_KEY`)
- [ ] Para cada entry em price-history, cruzar por `orcamento_id` + `escola`:
  - Se tem resultado "ganho" → `tipo: "ganho"`, `precoReal: valor_proposta`
  - Se tem resultado "perdido" → `tipo: "perdido"`, `precoVencedor: valor_vencedor`, `concorrente: fornecedor_vencedor`
  - Se sem resultado → `tipo: "proposta"` (default atual)
- [ ] Incluir campos no output: `tipo`, `precoVencedor`, `concorrente`, `motivoPerda`

### AC-2: summarize-price-history.js com estatísticas por tipo
- [ ] Gerar 3 blocos de estatísticas por SKU:
  - `skuSummary`: geral (como hoje)
  - `skuWonSummary`: apenas tipo="ganho" (min/median/max de preços vencedores)
  - `skuLostSummary`: apenas tipo="perdido" (min/median/max de preços perdedores)
- [ ] Adicionar campos em cada summary:
  - `totalGanhos`, `totalPerdidos`, `taxaConversao`
  - `medianPrecoGanho`, `medianPrecoPerdido`
  - `deltaGanhoPerdido`: diferença % entre preço médio ganho vs perdido
- [ ] Manter backward compatibility: `skuSummary` e `skuRegionSummary` continuam existindo

### AC-3: Validação de integridade
- [ ] Se resultado referencia orçamento que não existe em quotes → logar warning, não crashar
- [ ] Se quotes.json está vazio → gerar summary vazio, não erro
- [ ] Manter `uniqueKey` de dedup existente
