# Story 4.71 — Fix inconsistencia saldo portal vs dashboard (normalização descrições)

- **Epic:** EPIC-4 — GDP Caixa Escolar
- **Status:** Ready
- **Complexity:** S (Small)
- **Agent:** @dev
- **Depends on:** None
- **Priority:** Critical (bug em produção afetando dados do contrato Lauriston)

## Descrição

O dashboard GDP (`gdp-contratos-module.js`) e o portal escolar (`gdp-portal.html`) calculam `qtdEntregue` de formas diferentes, causando inconsistência no saldo exibido. Exemplo real: contrato Lauriston, cebola 100kg — dashboard mostra saldo 50kg (correto), portal mostra 100kg entregue (incorreto).

**Root Cause (identificado por @analyst):**
1. Portal não aplica `.trim()` na normalização de descrições ao agregar pedidos (linha ~990-993)
2. Portal não aplica `.trim()` no lookup de itens do contrato (linha ~1001)
3. Portal NÃO recalcula `qtdEntregue` a partir dos pedidos reais como o dashboard faz em `abrirContrato()` (linhas 2811-2825 de `gdp-contratos-module.js`)
4. Portal usa valor stale de `c.itens[].qtdEntregue` do localStorage, enquanto dashboard recalcula dinamicamente

## Acceptance Criteria

### AC1: Normalização de descrições no portal
- **Given** um contrato com item "Cebola " (com espaço trailing)
- **When** o portal agrega quantidades de pedidos para esse item
- **Then** a agregação aplica `.toUpperCase().trim()` tanto na key do pedido quanto no lookup do contrato
- **And** o match funciona mesmo com espaços extras ou diferenças de case

### AC2: Recálculo de qtdEntregue no portal
- **Given** o portal carrega um contrato com pedidos associados
- **When** monta a lista de itens para exibição (seção saldo/acompanhamento)
- **Then** recalcula `qtdEntregue` a partir dos pedidos reais (como o dashboard faz)
- **And** usa match por `itemNum` como critério primário e descrição normalizada como fallback

### AC3: Consistência dashboard vs portal
- **Given** um contrato visualizado no dashboard GDP e no portal escolar
- **When** ambos exibem saldo ou quantidade entregue
- **Then** os valores são idênticos (mesma lógica de cálculo)

## Scope

### IN
- Fix normalização `.trim()` em `gdp-portal.html` (agregação e lookup)
- Replicar lógica de recálculo de `qtdEntregue` do dashboard no portal
- Match por `itemNum` como critério primário

### OUT
- Refatoração do dashboard
- Mudanças no backend/API
- Mudanças no schema Supabase

## Análise Técnica (@analyst)

### Arquivo 1: `gdp-portal.html`
- **Linha ~990-993:** Agregação de pedidos — adicionar `.trim()` na key
- **Linha ~1001:** Lookup por descrição — adicionar `.trim()`
- **Linhas ~999-1029:** Replicar lógica de recálculo similar a `gdp-contratos-module.js:2811-2825`

### Arquivo 2: `gdp-contratos-module.js` (referência)
- **Linhas 2811-2825:** Lógica correta de recálculo que deve ser replicada no portal
