## Story

**As a** gestor do sistema GDP que usa múltiplos computadores,
**I want** que todas as alterações feitas em um computador apareçam automaticamente nos demais,
**so that** eu tenha dados consistentes em qualquer máquina que eu usar.

---

## Contexto / Histórico

Análise do @analyst revelou que o polling automático (`app-sync.js:281`) cobre apenas `pedidos` e `contratos`. As demais tabelas (`notas_fiscais`, `contas_receber`, `contas_pagar`, `entregas`) são gravadas no Supabase via `gdpApi.save()` mas NÃO são buscadas automaticamente em outros computadores.

---

## Acceptance Criteria

### AC-1 — Estender polling para todas as tabelas de negócio

**Arquivo:** `app-sync.js` — `pollForChanges()` (~linha 281)

- [ ] Adicionar `notas_fiscais`, `contas_receber`, `contas_pagar` e `entregas` ao array `tables`
- [ ] Manter a mesma lógica de merge: filtro por `empresa_id`, delta por `updated_at`, proteção de deletados
- [ ] Polling continua a cada 30 segundos
- [ ] Após merge, chamar `renderAll()` para atualizar a UI

### AC-2 — Verificar consistência do empresa_id

**Arquivo:** `gdp-api.js` — `getEmpresaId()` (~linha 31)

- [ ] Logar o `empresa_id` usado no console ao inicializar: `[Sync] empresa_id: XXXXX`
- [ ] Se `empresa_id` for o fallback `'LARIUCCI'`, mostrar aviso no console
- [ ] Garantir que o login (legacy e Supabase) sempre seta `syncUserId` em `nexedu.empresa`

---

## Technical Notes

### Arquitetura atual de sync:
- `gdpApi.save()` → grava no Supabase (tabela dedicada por entidade)
- `pollForChanges()` → busca delta por `updated_at` a cada 30s
- Polling atual: APENAS `['pedidos', 'contratos']`
- Faltam: `notas_fiscais`, `contas_receber`, `contas_pagar`, `entregas`

### Merge strategy:
- Last-write-wins por `updated_at`
- Itens deletados localmente são protegidos via `deletedIds` Set
- Após merge, `renderAll()` atualiza UI
