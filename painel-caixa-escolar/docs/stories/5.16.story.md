# Story 5.16 — Otimização de Performance do Supabase

**Epic:** 5 (Technical Debt v2)
**Sprint:** 5 (Performance)
**Status:** Draft
**Agente:** @data-engineer + @dev
**Prioridade:** Crítica
**Estimativa:** 8-12h

---

## Problema

O banco Supabase (Nano Compute) está em exaustão de recursos:
- RAM committed 1.59 GiB excede limite 1.20 GiB (132%)
- SWAP constante — banco usando disco como memória
- Disk I/O throttled — burst credits esgotados
- Resultado: timeouts 504/522, dados não sincronizam, sistema instável

Fonte: Supabase Support (2026-06-02) — diagnóstico com métricas.

## Objetivo

Reduzir consumo de RAM e I/O do banco para funcionar estável no plano atual (ou Micro), eliminando gargalos no código que geram carga desnecessária.

---

## Acceptance Criteria

### AC1: Reduzir canais Realtime (RAM)
- [ ] Reduzir WebSocket de 12 canais para 6 essenciais (contratos, pedidos, notas_fiscais, clientes, extratos, conciliacoes)
- [ ] Remover subscriptions de: sync_data, resultados_orcamento, radar_equivalencias, contas_receber, contas_pagar, entregas
- [ ] Contas/entregas: carregar sob demanda quando usuário acessa a aba (não no boot)
- [ ] Estimativa de economia: ~50% das conexões Realtime

### AC2: Reduzir audit triggers (Disk I/O)
- [ ] Desabilitar audit_trigger em tabelas de baixo risco: data_snapshots, audit_log (circular), extratos, conciliacoes
- [ ] Manter audit apenas em: contratos, pedidos, notas_fiscais, clientes (críticas)
- [ ] Estimativa de economia: ~40% das escritas de audit

### AC3: Otimizar syncToCloud (RAM + I/O)
- [ ] Aumentar debounce de syncToCloud de 2s para 30s
- [ ] Reduzir SYNC_KEYS para apenas keys que realmente mudam frequentemente
- [ ] Remover batch upsert no sync_data — usar writes individuais por key alterada
- [ ] Não sincronizar keys > 100KB (descartar silenciosamente)

### AC4: Otimizar queries do boot (CPU)
- [ ] Supabase-First boot: carregar tabelas em sequência (não Promise.all) para evitar pico de CPU
- [ ] Adicionar `select=` com colunas específicas em vez de `select=*` (reduz payload)
- [ ] Limitar a 500 registros por tabela no boot (paginar sob demanda)

### AC5: Limpar dados acumulados (Disk)
- [ ] Executar `run_retention_cleanup()` para limpar snapshots > 90 dias e audit > 365 dias
- [ ] Remover sync_data entries órfãos (keys que não existem mais no SYNC_KEYS)
- [ ] Verificar e limpar tabela `nexedu_sync` se existir dados legados

### AC6: Upgrade Nano → Micro (Infra)
- [ ] Fazer upgrade de compute no Supabase Dashboard (Settings → Compute)
- [ ] No Pro tier, Nano e Micro custam o mesmo — upgrade é grátis
- [ ] Verificar métricas pós-upgrade: RAM < 80%, SWAP = 0, EBS balance > 50%

---

## Métricas de Sucesso

| Métrica | Antes | Meta |
|---------|-------|------|
| RAM committed | 1.59 GiB (132%) | < 800 MiB (< 80%) |
| SWAP usage | Constante | Zero |
| EBS I/O balance | Burst credits esgotados | > 50% |
| Timeouts 504/522 | Frequentes | Zero |
| Canais Realtime | 12 | 6 |
| Audit triggers | 9 tabelas | 4 tabelas |
| Boot time | ~3-5s | < 2s |

## Ordem de Execução

1. **AC6** primeiro (upgrade Micro — impacto imediato, zero risco)
2. **AC5** segundo (limpeza de dados — libera disco)
3. **AC1** terceiro (reduzir Realtime — libera RAM)
4. **AC2** quarto (reduzir audit — libera I/O)
5. **AC3** + **AC4** por último (otimizar código — menor risco)

## Riscos

| Risco | Mitigação |
|-------|-----------|
| Reduzir Realtime pode atrasar sync de contas/entregas | Carregar sob demanda quando usuário abre a aba |
| Desabilitar audit perde rastreabilidade | Manter nas 4 tabelas mais críticas |
| Limpar sync_data pode perder dados auxiliares | Verificar keys antes de deletar |
