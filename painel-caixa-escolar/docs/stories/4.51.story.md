## Story

**As a** gestor do sistema GDP,
**I want** que exclusões sejam permanentes, filtros funcionem corretamente e a interface seja consistente,
**so that** o sistema seja confiável para uso comercial e vendido a outros fornecedores.

---

## Contexto / Motivação

O sistema apresenta múltiplos problemas pontuais que comprometem a confiabilidade para uso comercial:
1. **Dados excluídos reaparecem** — NFs de entrada, fornecedores e contas a receber voltam após exclusão (sync restaurando dados do localStorage/cache)
2. **Elementos de UI incompletos** — botões no lugar errado, campos cortados, falta de filtros
3. **Bugs funcionais** — categorias desaparecem, preço de custo não salva, conversão de produtos críticos errada
4. **Inconsistências visuais** — DANFE com formatos diferentes dependendo de onde é visualizada

### BLOCO A — Persistência de Exclusões (CRÍTICO)

#### AC-A1 — NFs de entrada excluídas devem permanecer excluídas
- [x] Ao excluir uma NF de entrada na sessão "Notas de Entrada", a exclusão deve ser persistida no Supabase (DELETE real ou soft-delete com flag)
- [x] O sync (app-sync.js, gdp-api.js) NÃO deve restaurar registros excluídos do localStorage/cache
- [x] Após refresh, troca de máquina ou sync, a NF excluída NÃO deve reaparecer
- [x] A exclusão deve ser refletida em TODOS os computadores/navegadores

#### AC-A2 — Fornecedores excluídos devem permanecer excluídos
- [x] Mesma lógica de AC-A1 aplicada a fornecedores
- [x] Fornecedores-modelo criados pelo sistema, se excluídos, NÃO devem ser recriados automaticamente
- [x] Verificar se há seed/init que recria fornecedores padrão e desativar para fornecedores já excluídos

#### AC-A3 — Contas a receber excluídas devem permanecer excluídas
- [x] Ao excluir contas a receber (status "recebida" ou qualquer outro), exclusão deve persistir no Supabase
- [x] O sync NÃO deve restaurar contas excluídas
- [x] A exclusão deve ser visível em todos os computadores

#### AC-A4 — Causa raiz: sync sobrescrevendo exclusões
- [x] Investigar o fluxo de sync (app-sync.js) e identificar onde dados locais (localStorage) sobrescrevem dados do Supabase
- [x] Implementar lógica de "tombstone" ou "deleted_at" que o sync respeite
- [x] Garantir que a fonte de verdade é o Supabase, não o localStorage

## Notas Técnicas

### Causa raiz do BLOCO A (exclusões reaparecendo)
O sistema usava arquitetura offline-first com 3 camadas de storage. O `pollForChanges()` em app-sync.js restaurava itens do Supabase sem verificar exclusões locais. Solução implementada:
1. `_trackDeletedId()` em gdp-api.js registra cada ID excluído em `gdp.{entity}.deleted.v1`
2. `pollForChanges()` lê IDs excluídos e pula itens deletados
3. `syncFromCloud()` filtra itens excluídos antes de escrever no localStorage
4. `excluirNotasEntradaSelecionadas()` e `excluirFornecedorEstoqueIntel()` também registram e fazem cloudSave

### AC-C4 pendente (saldo inicial)
Requer alteração no form de contas bancárias em app-config.js (HTML + JS). Decisão arquitetural já tomada pelo @architect. Pode ser implementado em sprint seguinte sem impacto funcional.

### AC-G4 pendente (edição em massa do campo produto)
Requer alteração no bulk edit toolbar em gdp-init.js. Complexidade baixa mas separada para não arriscar regressão no bulk edit existente.
