## Story

**As a** operador que envia pedidos ao Tiny/Olist,
**I want** que as unidades dos itens cheguem corretas ao pedido final no Tiny,
**so that** a NF-e é emitida com unidades certas (KG, LT, PCT) e não "UN" genérico.

---

## Root Cause (confirmado via code audit)

### Bug 1: `searchTinyProduct()` não retorna `unidade` (CRÍTICO)

Em `order.js` linhas 270-277, a função retorna `{codigo, ncm, nome, id}` mas **omite `unidade`**:

```javascript
// ATUAL (bugado) — order.js:271-276
return {
  codigo: bestProduct.codigo || "",
  ncm: bestProduct.ncm || "",
  nome: bestProduct.nome || "",
  id: bestProduct.id || "",
  // ← FALTA: unidade: bestProduct.unidade || ""
};
```

Na linha 420 do Step 1:
```javascript
item.unidade = existing.unidade || item.unidade;
//             ^^^^^^^^^^^^^^^^ = SEMPRE undefined!
```

**Resultado:** `existing.unidade` é sempre `undefined` → cai no fallback `item.unidade` → que pode ser "UN" genérico ou o valor não-normalizado do frontend.

### Bug 2: API `produtos.pesquisa.php` pode não retornar `unidade`

A API de pesquisa do Tiny retorna campos básicos (`codigo`, `nome`, `ncm`, `id`). O campo `unidade` pode não estar disponível na resposta de pesquisa. Nesse caso, é necessário uma chamada secundária a `produto.obter.php` com o `id` do produto para obter os detalhes completos incluindo `unidade`.

### Bug 3: `normalizeUnit()` fallback silencioso para unidades desconhecidas

Linha 218: `return map[u] || u.slice(0, 3);`
- "GARRAFA" → "GAR" (inválido no Tiny)
- "FARDO" → "FAR" (inválido no Tiny)
- "SACHÊ" → "SAC" (confunde com "SACO"→"SC")
- Não há log/warning quando uma unidade desconhecida é truncada

### Impacto

Pedidos chegam ao Tiny com unidade "UN" para itens que deveriam ser KG, LT, PCT, etc. Isso afeta:
- Emissão de NF-e (unidade errada na nota)
- Controle de estoque (peso vs unidade)
- Compliance fiscal

### AC-2: `normalizeUnit()` cobre mais unidades e loga desconhecidas

- [x] Se detectar inconsistência: mostrar warning toast (não bloquear envio)

- [x] 1.2 Testar se `produtos.pesquisa.php` retorna `unidade` na resposta:
  - Se SIM: fix completo com 1.1
  - Se NÃO: implementar fallback com `produto.obter.php`:
  ```javascript
  // Após encontrar bestProduct, se unidade não veio na pesquisa:
  if (bestProduct && !bestProduct.unidade && bestProduct.id) {
    const detailForm = new URLSearchParams();
    detailForm.set("token", token);
    detailForm.set("formato", "json");
    detailForm.set("id", bestProduct.id);
    try {
      const detResp = await fetch("https://api.tiny.com.br/api2/produto.obter.php", {
        method: "POST",
        headers: { "Content-Type": "application/x-www-form-urlencoded" },
        body: detailForm.toString(),
      });
      const detParsed = JSON.parse(await detResp.text());
      const fullProd = detParsed.retorno?.produto;
      if (fullProd) {
        bestProduct.unidade = fullProd.unidade || "";
      }
    } catch (_e) { /* best-effort */ }
  }
  ```
