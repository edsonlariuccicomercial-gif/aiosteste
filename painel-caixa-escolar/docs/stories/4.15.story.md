## Story

**As a** operador que gerencia contratos e emite NF-e pelo Tiny,
**I want** um banco de produtos limpo, importado do Tiny e com opcao de enviar novos produtos ao ERP,
**so that** o banco de produtos reflita exatamente o cadastro do Tiny, eliminando divergencias na emissao de notas fiscais.

---

## Contexto / Motivacao

A estrategia anterior de sincronizacao automatica de produtos com o Tiny (Stories 4.14, tentativas anteriores) gerou problemas:
- SKUs sobrescritos incorretamente
- Unidades perdidas no fluxo
- Cadastros duplicados no Tiny

**Nova estrategia:** O banco de produtos (`gdp.produtos.v1`) sera a fonte da verdade local, alimentado por importacao manual da lista do Tiny. Novos produtos sao criados no banco e enviados ao Tiny sob demanda.

> **Nota:** A tela de sincronizacao na criacao de contrato (comparacao itens contrato vs banco) sera implementada na Story 4.16 como fase seguinte.

## Acceptance Criteria

### AC-1: Limpeza total do Banco de Produtos (resolver 21 itens remanescentes)

- [ ] Implementar funcao `limparTodoBancoProdutos()` que exclui TODOS os itens de `gdp.produtos.v1` (localStorage E Supabase cloud sync)
- [ ] Botao "Limpar Tudo" visivel na aba Banco de Produtos (com confirmacao dupla: confirm + digitar "LIMPAR")
- [ ] Apos limpeza, o contador do tab deve mostrar 0 e a tabela deve mostrar estado vazio
- [ ] Investigar e corrigir se o cloud sync (Supabase) esta re-adicionando itens deletados localmente — garantir que a exclusao local propague para o cloud

### AC-2: Importacao de produtos do Tiny para o Banco

- [ ] Botao "Importar do Tiny" na aba Banco de Produtos
- [ ] Ao clicar, chamar API Tiny `produtos.pesquisa.php` paginada (paginas 1 ate N) para trazer todos os produtos
- [ ] Para cada produto importado, extrair e salvar: `descricao` (nome), `sku` (codigo), `ncm`, `unidade`
- [ ] Preview dos produtos encontrados antes de confirmar importacao (tabela com checkbox para selecionar quais importar)
- [ ] Produtos ja existentes no banco (match por SKU) sao atualizados, nao duplicados
- [ ] Produtos novos sao inseridos com `id` gerado (`PROD-{timestamp}-{random}`)
- [ ] Mostrar progresso da importacao (ex: "Carregando pagina 3... 150 produtos encontrados")

### AC-3: Botao "Enviar ao ERP" no modal de Novo Produto

- [ ] No modal `modal-produto` (funcao `novoProdutoManual()`), adicionar botao "Enviar ao ERP" ao lado do botao "Salvar"
- [ ] O botao "Enviar ao ERP" deve: (1) salvar o produto no banco local, (2) enviar ao Tiny via `POST /api/tiny-produtos` com action `cadastrar`
- [ ] Campos enviados ao Tiny: `descricao`, `unidade`, `ncm`, `codigo` (SKU se preenchido, senao auto-gerar)
- [ ] Feedback visual: spinner durante envio, toast de sucesso/erro, SKU retornado do Tiny atualizado no banco local
- [ ] Se o produto ja existe no Tiny (match por nome/SKU), mostrar aviso e perguntar se deseja atualizar ou cancelar
- [ ] Botao desabilitado se `descricao` estiver vazio (validacao minima)

### Problema dos 21 Itens

Os 21 itens que nao foram excluidos provavelmente persistem porque:
1. O cloud sync (Supabase) tem uma copia antiga com esses itens
2. Ao fazer sync, o cloud sobrescreve a versao local (conflict resolution: `cloud.updated_at > local.updatedAt`)
3. **Fix:** A funcao `limparTodoBancoProdutos()` deve setar `updatedAt` para `new Date().toISOString()` ANTES de salvar, garantindo que a versao limpa seja mais recente que qualquer copia no cloud

### API Tiny — Listagem Paginada

```
POST https://api.tiny.com.br/api2/produtos.pesquisa.php
  token={TINY_TOKEN}&formato=json&pagina=1

Resposta: { retorno: { status: "OK", pagina: 1, numero_paginas: N, produtos: [...] } }

Cada produto: { produto: { id, nome, codigo, ncm, unidade, preco, ... } }
```
