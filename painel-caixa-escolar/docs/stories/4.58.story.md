## Story

**As a** gestor do sistema GDP,
**I want** que o extrato que já importei e conciliei apareça na aba Conciliação Bancária em formato fechado, e que o PDF da NF-e tenha dados do emitente mesmo quando preview não existe,
**so that** eu não perca meu trabalho de conciliação e os emails com NF sejam enviados com dados completos.

---

## Contexto / Histórico

### Bug 1: Extrato não aparece na Conciliação
Teste Playwright revelou que:
- `gdp.conciliacao.v1` tem 13 itens (10 conciliados) — dados existem
- `gdp.extratos.v1` tem 1 extrato (do teste Playwright, não o original do usuário)
- O `getCaixaResumo()` (gdp-pedidos.js:1782) usa `loadConciliacao()` — mostra no Caixa
- Mas a tabela colapsada só aparece se `loadExtratos()` retorna extratos
- **No browser do usuário**: itens de conciliação existem mas `gdp.extratos.v1` foi zerado pelos bugs 4.55/4.56

**Fix:** No boot, se existem itens em `gdp.conciliacao.v1` mas zero extratos em `gdp.extratos.v1`, reconstruir o extrato automaticamente.

### Bug 2: PDF NF em branco para NFs sem preview
Teste Playwright revelou:
- 30 NFs autorizadas, 28 têm `sefaz.preview.emitente/destinatario`
- **2 NFs (1433, 1447) NÃO têm `sefaz.preview`** — PDF será branco
- Fallback para `nf.cliente` funciona para destinatário
- **Falta fallback para emitente** — sem preview, emitente fica `{}`

**Fix:** Carregar dados do emitente de `nexedu.empresa` (localStorage) como fallback.

## Acceptance Criteria

### AC-1 — Reconstruir extrato quando conciliação existe mas extratos zerados

**Arquivo:** `gdp-core.js` — `initializeData()` (boot sequence, ~linha 811)

- [ ] Após a migração de extratoId (já existente), adicionar verificação: se `loadConciliacao().length > 0` E `loadExtratos().length === 0`, reconstruir extrato
- [ ] Reconstrução: criar 1 extrato com dados derivados dos itens de conciliação existentes
- [ ] Dados do extrato reconstruído:
  - `id`: 'ext-recovered-' + Date.now()
  - `data`: data do item mais antigo (ou hoje)
  - `arquivo`: 'Extrato recuperado (conciliação existente)'
  - `contaFinanceira`: extrair da config contas-bancarias (mesma lógica de importarExtratoBancario)
  - `conciliados`: count de items com `conciliado === true`
  - `total`: count total de items
  - `isOpen`: false
- [ ] Vincular todos os itens sem `extratoId` ao extrato reconstruído
- [ ] Salvar com `saveExtratos()` e `saveConciliacao()`
- [ ] Log: `[boot] Recovered extrato from N existing conciliacao items`
- [ ] Após reconstrução, extrato deve aparecer colapsado na tabela

**Teste de validação (Playwright):**
1. Inserir itens em `gdp.conciliacao.v1` via evaluate (simular dados do usuário)
2. Zerar `gdp.extratos.v1`
3. Reload página
4. Verificar que extrato reconstruído aparece na tabela colapsada
5. Clicar → deve expandir com lançamentos corretos

---

### AC-2 — Fallback emitente no DANFE para NFs sem preview

**Arquivo:** `gdp-notas-fiscais.js` — `gerarDanfeHtmlCompleto()` (linha ~1597)

- [ ] Se `preview.emitente` estiver vazio, carregar dados de `nexedu.empresa` (localStorage) como fallback
- [ ] Dados de fallback do emitente: razaoSocial, cnpj, endereco (logradouro, numero, bairro, cidade, uf, cep, telefone)
- [ ] Se `preview.destinatario` estiver vazio E `nf.cliente` existir, montar destinatário com: nome, cnpj, endereco (se disponível)
- [ ] PDF NUNCA deve ser gerado com todos os campos vazios — se nem preview nem fallback existem, mostrar toast e não enviar

**Teste de validação:**
1. NF autorizada COM preview → PDF com dados completos (já funciona)
2. NF autorizada SEM preview (ex: NF-1433) → PDF com dados do fallback nexedu.empresa + nf.cliente
