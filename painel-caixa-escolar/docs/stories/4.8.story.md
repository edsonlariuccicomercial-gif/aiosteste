## Story

**As a** fornecedor que sincroniza itens do contrato com o Banco de Produtos e envia pedidos ao Olist,
**I want** que o SKU seja propagado corretamente em toda a cadeia (Contrato ↔ Banco ↔ Pedido → Olist),
**so that** os pedidos chegam ao ERP com o SKU correto e não há divergência entre o que está no sistema e o que chega no Olist.

---

## Design Decision (aprovada pelo PO)

**Hierarquia de fonte de verdade do SKU:** `Tiny/Olist > Banco de Produtos > Item do Contrato`

- Se Tiny retorna SKU → propagar para contrato + banco
- Se Banco tem SKU (por match de descrição) → usar no pedido e persistir no contrato
- Item do contrato só é fonte se as outras duas estiverem vazias

## Acceptance Criteria

### AC-1: Catálogo do fornecedor persiste SKU enriquecido no item do contrato
- [ ] Em `abrirCatalogoEscolar()`: quando o match com Banco de Produtos encontra um SKU (`bp.sku`), **salvar de volta** no item do contrato: `item.sku = bp.sku`
- [ ] Chamar `saveContratos()` após enriquecer os itens com SKU do Banco
- [ ] O enriquecimento deve rodar ANTES de renderizar a tabela do catálogo

### AC-2: Pedido do catálogo inclui SKU nos itens
- [ ] Em `criarPedidoCatalogo()`: ao montar `itensP`, incluir campo `sku` extraído do item do contrato (que agora está enriquecido pelo AC-1): `sku: item.sku || ''`
- [ ] Em `salvarPedidoManual()`: mesmo fix — incluir `sku` nos itens do pedido

### AC-3: Envio ao Olist usa SKU enriquecido com fallback em cadeia
- [ ] Em `enviarPedidoOlist()`: a resolução de SKU deve seguir a hierarquia:
  1. `ctrItem.sku` (item do contrato — já enriquecido pelo Banco/Tiny)
  2. Fallback: buscar no Banco de Produtos por descrição normalizada
  3. Fallback final: `i.sku` do pedido ou string vazia
- [ ] Logar no console quando SKU está vazio ao enviar: `console.warn("[Olist] SKU vazio para item:", i.descricao)`

### AC-4: Retorno do Tiny propaga SKU para Contrato + Banco
- [ ] Confirmar que o handler de resposta do Tiny (`cadastrarTiny`/`enviarTiny`) já salva `item.sku = r.sku` no contrato (linha ~5206)
- [ ] **Adicionar**: quando Tiny retorna SKU, também atualizar/inserir no Banco de Produtos (`gdp.produtos.v1`) — chamar `adicionarAoBancoProdutos` com o SKU atualizado
- [ ] Garantir que `saveContratos()` e `saveBancoProdutos()` são chamados após a propagação

### AC-5: `adicionarAoBancoProdutos` preserva SKU existente do Banco
- [ ] Na função `adicionarAoBancoProdutos()`: se o produto já existe no Banco e **já tem SKU**, NÃO sobrescrever com string vazia do item do contrato
- [ ] Lógica: `sku: item.sku || existing.sku || ''` (preservar o mais confiável)

### AC-6: `cadastrarTinyItem` envia SKU existente ao Tiny
- [ ] Em `cadastrarTinyItem()`: incluir `sku: item.sku || ''` no payload enviado ao `/api/tiny-produtos`
- [ ] Se o item já tem SKU (do Banco ou de envio anterior), o Tiny recebe e pode fazer match ao invés de criar duplicata

## Tasks / Subtasks

### Task 1: Enriquecer e persistir SKU no catálogo (AC-1)

- [ ] 1.1 Em `abrirCatalogoEscolar()` (~linha 3013): após o loop de enrichment dos itens, adicionar bloco para persistir SKU:
  ```javascript
  let skuUpdated = false;
  c.itens.forEach((item, idx) => {
    const descNorm = (item.descricao || '').toUpperCase().trim();
    let bp = null;
    if (item.sku) bp = _bp.find(p => p.sku && p.sku.toUpperCase() === item.sku.toUpperCase());
    if (!bp) bp = _bp.find(p => (p.descricao || '').toUpperCase().trim() === descNorm);
    if (!bp) {
      const words = descNorm.split(/\s+/).slice(0, 3).join(' ');
      if (words.length > 5) bp = _bp.find(p => (p.descricao || '').toUpperCase().trim().startsWith(words));
    }
    if (bp && bp.sku && !item.sku) {
      item.sku = bp.sku;
      skuUpdated = true;
    }
  });
  if (skuUpdated) saveContratos();
  ```

### Task 3: Fallback em cadeia no envio ao Olist (AC-3)

- [ ] 3.1 Em `enviarPedidoOlist()`: alterar resolução de SKU:
  ```javascript
  const items = p.itens.map(i => {
    const ctrItem = ctr ? ctr.itens.find(ci => ci.num === i.itemNum || ci.descricao === i.descricao) : null;
    // Hierarquia: Contrato (enriquecido) > Pedido > Banco de Produtos > vazio
    let sku = (ctrItem && ctrItem.sku) || i.sku || '';
    if (!sku && ctr) {
      // Fallback: buscar no Banco de Produtos por descrição
      const bpItens = (() => { try { return JSON.parse(localStorage.getItem('gdp.produtos.v1'))?.itens || []; } catch(_) { return []; } })();
      const descNorm = (i.descricao || '').toUpperCase().trim();
      const bpMatch = bpItens.find(bp => (bp.descricao || '').toUpperCase().trim() === descNorm);
      if (bpMatch && bpMatch.sku) sku = bpMatch.sku;
    }
    if (!sku) console.warn("[Olist] SKU vazio para item:", i.descricao);
    return { sku, description: i.descricao, name: i.descricao, qty: i.qtd, unitPrice: i.precoUnitario, itemNum: i.itemNum };
  });
  ```

## Dev Notes

### Hierarquia de SKU (decisão de produto)

```
Tiny/Olist ERP (fonte primária)
       ↓ retorna SKU
Item do Contrato (recebe e persiste)
       ↓ propaga
Banco de Produtos ERP (recebe e persiste)
       ↓ enriquece
Catálogo do Fornecedor (display + persist back)
       ↓ herda
Pedido (itens com SKU)
       ↓ envia
Olist API (recebe SKU correto)
```

### Pontos de divergência encontrados (diagnóstico River)

| # | Local | Problema |
|---|-------|----------|
| 1 | Import de mapa → contrato | Não extrai SKU (sem campo no mapa) |
| 2 | Form +Item | Não tem campo SKU |
| 3 | `adicionarAoBancoProdutos` | Puxa `item.sku` vazio do contrato |
| 4 | `abrirCatalogoEscolar` | Mostra SKU do Banco mas NÃO persiste |
| 5 | `criarPedidoCatalogo` | Não inclui SKU nos itens do pedido |
| 6 | `enviarPedidoOlist` | SKU do contrato vazio → envia vazio |
| 7 | `cadastrarTinyItem` | Não envia SKU existente ao Tiny |

### Funções a modificar

- `abrirCatalogoEscolar()` — persistir SKU enriquecido
- `criarPedidoCatalogo()` — incluir SKU nos itens

- `adicionarAoBancoProdutos()` — preservar SKU existente
- `cadastrarTinyItem()` — enviar SKU existente
