# Story 2.10 — Consolidacao de Dados e Performance

**Epic:** 2 — Resolucao de Debito Tecnico
**Sprint:** 5 (Semana 10-12)
**Prioridade:** P3 — MEDIO
**Estimativa:** 38 horas
**Debitos resolvidos:** D-05, D-08, D-10, D-12, D-13, D-14, D-15, D-21, D-22, D-24, SYS-6, SYS-9

---

## Descricao

Consolidar a camada de dados: unificar pipeline de coleta (eliminar duplicidade Playwright/REST), implementar backup automatico, transacoes multi-arquivo, versionamento de dados, logging estruturado e health monitoring. Tambem otimizar performance do scan SGD e implementar paginacao.

---

## Criterios de Aceitacao

- [ ] Pipeline de coleta unificado (apenas REST, Playwright eliminado ou reservado para fallback)
- [ ] Backup automatico de JSON files com retencao de 30 dias
- [ ] Transacoes multi-arquivo implementadas (write-ahead log)
- [ ] Registros tem `updatedAt` e changelog de mudancas
- [ ] Sistema de migracao de schema JSON funcional
- [ ] Estimativa de custo baseada em banco-precos.json (nao mais hash ficticio)
- [ ] Cache de expense groups por rede SGD
- [ ] Scan SGD paralelizado (pool de 5 requests concorrentes)
- [ ] Tempo de scan < 2 minutos para 60 orcamentos
- [ ] Paginacao server-side implementada (default 50 items/page)
- [ ] Logging estruturado com nivel, timestamp, contexto
- [ ] Health check endpoint `GET /api/health` com status de todas as dependencias
- [ ] Alertas de falha configurados (scan, sync, API down)

---

## Tarefas

### Unificacao de Coleta

- [ ] Analisar `collect-sgd-orcamentos.js` (Playwright) vs `executeSgdScan()` (REST)
- [ ] Unificar em pipeline unico REST (via `sgd-client.js`)
- [ ] Manter Playwright como fallback documentado (quando REST falha)
- [ ] Eliminar `quotes.json` como duplicacao de `orcamentos.json`
- [ ] Implementar reconciliacao de dados entre fontes

### Backup e Transacoes

- [ ] Criar modulo `backup.js` — snapshot timestamped de JSON files
- [ ] Implementar rotina de backup diaria (antes do scan)
- [ ] Configurar retencao: manter ultimos 30 backups, excluir mais antigos
- [ ] Considerar Netlify Blobs para backup remoto
- [ ] Implementar `transaction.js` — write-ahead log para operacoes multi-arquivo
- [ ] Transacao: orcamentos.json + scan-log.json escritos atomicamente

### Versionamento e Schema

- [ ] Adicionar `updatedAt` em todos os registros
- [ ] Implementar changelog simplificado (ultimas 10 mudancas por registro)
- [ ] Criar `migration.js` — sistema de migracao de schema
- [ ] Migracoes: v1 (atual) → v2 (com updatedAt, campos novos)
- [ ] Auto-migrate ao iniciar o server

### Estimativas Reais

- [ ] Refatorar calculo de `custoEstimado` e `precoSugerido`
- [ ] Integrar com `banco-precos.json` — match por grupo de despesa
- [ ] Calcular: custo base + margem padrao + frete estimado
- [ ] Quando nao houver match no banco, usar media historica de `quotes.json`

### Performance

- [ ] Paralelizar scan SGD com pool de 5 requests concorrentes
- [ ] Implementar rate limiting para API SGD (respeitar limites gov)
- [ ] Cache de expense groups por networkId (evitar re-fetch)
- [ ] Paginacao server-side: `GET /api/orcamentos?page=1&limit=50`
- [ ] Migrar `readFileSync` restantes para `fs.promises`

### Observabilidade

- [ ] Criar modulo `logger.js` — logging estruturado (nivel, timestamp, contexto)
- [ ] Niveis: debug, info, warn, error
- [ ] Formato: JSON (parseable por ferramentas)
- [ ] Rotacao de logs (D-18): max 1000 entradas ou 5MB por arquivo
- [ ] Endpoint `GET /api/health` — status do server, ultima varredura, conectividade SGD
- [ ] Health check inclui: disk space, JSON files validos, API SGD acessivel

## Dependencias

- **Story 2.3** — Validacao schema (Zod) e safe-json implementados
- **Story 2.2** — Auth nas APIs implementada
- **Story 2.1** — Lock file de concorrencia implementado

## Riscos

- Eliminacao do Playwright pode perder cenarios edge — manter como fallback
- Paralelismo no scan pode sobrecarregar API SGD — rate limiting conservador
- Migracao de schema pode falhar em dados existentes — modo "best effort"
