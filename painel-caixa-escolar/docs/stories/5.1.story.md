# Story 5.1 — Migrar RLS para auth.uid() exclusivo

**Epic:** 5 (Technical Debt v2)
**Sprint:** 1 (Segurança)
**Status:** Draft
**Agente:** @data-engineer
**Débitos:** TD-C1, TD-A7
**Estimativa:** 20-32h

---

## Objetivo

Migrar todas as RLS policies de "anon full access" (permissivo) para auth.uid() exclusivo, garantindo isolamento real entre empresas no banco de dados.

## Contexto

Atualmente, as policies da Migration 007 (`anon_full_access`, `auth_full_access`) usam `USING(true) WITH CHECK(true)`, o que efetivamente desabilita o isolamento multi-tenant. Qualquer requisição com a anon key pode ler/escrever dados de qualquer empresa.

## Acceptance Criteria

- [ ] AC1: Remover todas as policies `anon_full_access` e `auth_full_access` da Migration 007
- [ ] AC2: Atualizar policies para usar `get_user_empresa_id()` (Migration 009) como filtro primário
- [ ] AC3: Garantir que portal de escolas continua funcionando (usa anon key + empresa_id específico)
- [ ] AC4: Criar migration 019 com as novas policies
- [ ] AC5: Testar isolamento: user A não deve ver dados de user B
- [ ] AC6: Extratos/conciliacoes (Migration 018) devem usar mesma estratégia
- [ ] AC7: Documentar estratégia RLS final no SCHEMA.md

## Out of Scope

- Migração de auth de escolas (Story 5.2 separada)
- Mudanças no frontend (apenas DB layer)

## Arquivos Impactados

- `supabase/migrations/019_rls_consolidation.sql` (novo)
- `docs/architecture/SCHEMA.md` (atualizar seção RLS)

## Riscos

- Frontend pode quebrar se policies bloquearem requests legítimos
- Portal escolar usa anon key — precisa de tratamento especial
