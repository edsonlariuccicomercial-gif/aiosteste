# Story 2.3 — Integridade de Dados e Validacao Schema

**Epic:** 2 — Resolucao de Debito Tecnico
**Sprint:** 1 (Semana 2-3)
**Prioridade:** P1 — URGENTE
**Estimativa:** 21 horas
**Debitos resolvidos:** D-06, D-07, D-09, SYS-4, SEC-7

---

## Descricao

Implementar tres pilares de integridade de dados: (1) escrita atomica de JSON para prevenir corrupcao, (2) validacao de schema com Zod em todos os pontos de entrada, e (3) matching deterministico de itens no envio ao SGD para evitar mapeamento incorreto de precos.

O debito D-09 e o mais urgente — o fuzzy match atual pode estar enviando propostas com precos errados ao sistema governamental.

---

## Criterios de Aceitacao

- [ ] Escrita de JSON usa padrao write-to-temp + rename atomico
- [ ] Schemas Zod definidos para: Orcamento, Proposta, Pedido, Preco, Escola
- [ ] Todos os endpoints validam input com Zod antes de processar
- [ ] Dados da API SGD sao validados antes de salvar em JSON
- [ ] Matching de itens no submit usa `idBudgetItem` (deterministico)
- [ ] Fallback de matching usa Levenshtein com threshold >= 80%
- [ ] Match com confianca < 90% requer confirmacao manual (flag no payload)
- [ ] Testes unitarios para: escrita atomica, validacao schema, matching itens
- [ ] Dados malformados sao rejeitados com erro descritivo (nao silenciosos)

## Dependencias

- **Nenhuma** — pode ser feito em paralelo com Story 2.2
- **npm:** `zod`, `fastest-levenshtein` (ou similar)

## Riscos

- Validacao Zod pode rejeitar dados historicos que nao conformam — definir modo "warn" para dados existentes
- Levenshtein pode ser lento com muitos itens — testar performance com 343 itens (ARP)
