# Story 5.15 — Migrar Central de Produtos para tabela dedicada Supabase

**Epic:** 5 (Technical Debt v2)
**Sprint:** 5 (Data Integrity)
**Status:** Draft
**Agente:** @data-engineer + @dev
**Estimativa:** 8-12h

---

## Objetivo

Migrar a Central de Produtos (`intel.central-produtos.v2`) e Custos Fornecedores (`intel.custos-fornecedores.v1`) de sync_data (última escrita ganha) para tabelas dedicadas no Supabase, eliminando divergência entre máquinas.

## Contexto

A Central de Produtos hoje usa sync_data como armazenamento. Cada máquina tem sua cópia no localStorage e a última que faz sync sobrescreve o cloud inteiro. Isso causa:
- Divergência de contagens entre máquinas (186/14 vs 376/10 vs 385/15)
- Perda de dados quando uma máquina com versão antiga faz sync
- Impossibilidade de edição concorrente (duas pessoas editando = última ganha)

Tabelas `produtos` e `estoque_simples` já existem no Supabase (migration 014) mas estão **vazias** — nunca foram populadas.

## Acceptance Criteria

- [ ] AC1: Migrar dados de `intel.central-produtos.v2` (sync_data) para tabela `produtos`
- [ ] AC2: Migrar dados de `intel.custos-fornecedores.v1` (sync_data) para nova tabela ou coluna em `produtos`
- [ ] AC3: Criar `gdpApi.produtos` e `gdpApi.custos_fornecedores` na API pública do gdp-api.js
- [ ] AC4: Incluir `produtos` no Supabase-First boot (gdp-init.js)
- [ ] AC5: Incluir `produtos` no gdp-realtime.js WebSocket subscriptions
- [ ] AC6: Remover `intel.central-produtos.v2` e `intel.custos-fornecedores.v1` do SYNC_KEYS
- [ ] AC7: Limpar entries duplicados no sync_data
- [ ] AC8: Dados consistentes entre todas as máquinas após migração
- [ ] AC9: Edições individuais de produto persistem sem sobrescrever todo o bloco

## Arquivos Impactados

- `supabase/migrations/026_migrate_central_produtos.sql` (novo — DML migration)
- `squads/caixa-escolar/dashboard/gdp-api.js` (expor produtos na API)
- `squads/caixa-escolar/dashboard/js/gdp-init.js` (incluir no boot)
- `squads/caixa-escolar/dashboard/js/gdp-realtime.js` (incluir no WebSocket)
- `squads/caixa-escolar/dashboard/app-state.js` (remover do SYNC_KEYS)

## Riscos

- Tabela `produtos` (migration 014) pode ter schema diferente do `intel.central-produtos.v2`
- Custos fornecedores são array aninhado — pode precisar de tabela separada
- Migração DML precisa rodar uma vez sem re-executar

## Referências

- Migration 014: `014_gdp_simplification.sql` (criou tabela `produtos` + `estoque_simples`)
- `app-state.js`: `loadCentralProdutos()`, `saveCentralProdutos()`, `CENTRAL_PRODUTOS_KEY`
- Schema `produtos`: id, empresa_id, descricao, sku, ncm, unidade, marca, grupo, produto_critico, custo_base, preco_referencia, margem_alvo
