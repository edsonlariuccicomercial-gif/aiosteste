## Story

**As a** operador que gerencia contratos no GDP,
**I want** que as funcionalidades de vincular produto, gerar demanda, confirmar demanda, estoque e lista de compras estejam no módulo GDP (gdp-contratos.html) onde pertencem logicamente,
**so that** o fluxo pós-licitação fique completo no GDP e não misturado com o módulo pré-licitação.

---

## Contexto / Motivação

As Stories 4.35 (Equivalências), 4.36 (Conversão na Demanda) e 4.37 (Estoque/Compras) foram implementadas em `app.js`/`index.html` (módulo pré-licitação/Radar). Porém esse fluxo é pós-licitação e pertence ao GDP (`gdp-contratos.html`). O GDP já tem módulos nativos de Contratos, Demandas, Estoque e Compra Inteligente que devem ser integrados.

**O que migrar:**
1. "Vincular Produto" (equivalências) → mover para a tela de detalhe do Contrato no GDP
2. "Gerar Demanda" (conversão) → integrar com o fluxo de Pedidos → Demandas no GDP
3. "Confirmar Demanda" → integrar com Inteligência → Demandas existente no GDP
4. "Estoque" e "Lista de Compras" → já existem no GDP (Inteligência → Estoque e Compra Inteligente) — integrar dados
5. Remover seções duplicadas do `index.html` (demandas-section, estoque-section, compras-section)

**Data layer permanece igual** (localStorage keys `gdp.equivalencias.v1`, `gdp.conversoes.v1`, `gdp.demandas.v1`, `gdp.estoque.v1`, `gdp.lista-compras.v1`) — apenas a UI e a lógica de render são movidas.

**Funções em `app.js` que precisam mover para `gdp-contratos.html`:**
- `getEquivalencia()`, `setEquivalencia()`, `getProdutoBySku()`, `loadEquivalencias()`, `saveEquivalencias()`
- `converterDemanda()`, `gerarDemanda()`, `confirmarDemanda()`
- `vincularProduto()`, `abrirModalVincular()`, `selecionarProdutoVincular()`, `criarEVincularProduto()`
- `renderDemandas()`, `renderEstoque()`, `renderListaCompras()`, `lancamentoEstoque()`
- `exportarListaCompras()`, `imprimirListaCompras()`

**Onde colocar em `gdp-contratos.html`:**
- Aba Contratos → na tela de detalhe do contrato, exibir itens com botão "Vincular Produto"
- Inteligência → sub-view Demandas → exibir demandas geradas a partir de pedidos, com botão confirmar
- Inteligência → Estoque → integrar dedução de estoque a partir de demandas confirmadas
- Inteligência → Compra Inteligente → integrar lista de compras a partir de demanda não atendida

**O que fica em `app.js`:**
- Funções de data layer (load/save) — compartilhadas entre módulos
- Funcionalidades "Gerar Contrato Unificado" e "Registrar como Ganho" (ponte pré→pós)

## Dev Notes

### Separação de responsabilidades: data layer vs UI

O data layer (funções de load/save) fica em `app.js` como camada compartilhada. As funções de render e interação UI são movidas para `gdp-contratos.html`. Para que `gdp-contratos.html` acesse o data layer, ele pode:
- Ler/escrever diretamente no `localStorage` (preferido — sem dependência de `app.js`)
- Ou incluir `app.js` como script (apenas se for viável sem side effects)

Recomendar a abordagem de `localStorage` direto para manter os módulos desacoplados.

### Chaves localStorage (não alterar)

```javascript
const KEYS = {
  equivalencias: 'gdp.equivalencias.v1',
  conversoes:    'gdp.conversoes.v1',
  demandas:      'gdp.demandas.v1',
  estoque:       'gdp.estoque.v1',
  listaCompras:  'gdp.lista-compras.v1',
};
```

### Padrão de integração na sub-view Demandas

```javascript
// Ao carregar a sub-view Demandas no GDP
function renderDemandasGDP() {

```

### Remoção de seções do index.html

Antes de remover, verificar se há event listeners em `app.js` apontando para IDs dessas seções (ex: `document.getElementById('demandas-section')`). Remover ou neutralizar esses listeners para evitar erros JavaScript silenciosos.
